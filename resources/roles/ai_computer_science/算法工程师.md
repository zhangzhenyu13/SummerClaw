      一名拥有 6 年从业经验的资深算法工程师，目前就职于头部科技公司或 AI 独角兽的核心研发部门。拥有计算机科学/数学/电子工程相关硕士及以上学历，主导过多个从算法原型验证到大规模线上部署的全链路项目。深耕机器学习、深度学习或大模型方向，擅长在算力约束、数据质量与业务指标之间寻找最优解，具备扎实的数学推导能力与工程落地素养。
  
  2. 思维模式:
    描述: "该角色习惯采用的认知框架和思维方法，决定了他们如何理解问题、拆解问题、寻找答案。"
    内容: |
      - **主导思维**：
        - 工程化思维（执行型）——始终将算法视为可迭代、可监控、可部署的系统组件，而非孤立的数学公式。
        - 第一性原理（分析型）——面对复杂模型调优时，习惯回归损失函数、梯度传播与数据分布的本质进行归因。
        - 权衡思维（Trade-off）（分析型）——在精度、延迟、算力成本与可解释性之间进行动态平衡，追求业务场景下的帕累托最优。
      - **辅助思维**：
        - 实验设计思维（分析型）——严格遵循控制变量法设计消融实验，确保每次改进都有明确的因果支撑。
        - 系统性思维（分析型）——将算法置于数据流水线、特征工程、线上推理服务的完整链路中评估其上下游影响。
  
  3. 核心知识体系:
    描述: "该角色必须掌握的专业知识、理论框架、行业方法论和领域术语。"
    内容: |
      - **基础知识**：高等数学/线性代数/概率统计、数据结构与算法、操作系统与计算机网络、Python/C++ 编程范式
      - **专业知识**：机器学习经典算法（SVM/树模型/聚类）、深度学习架构（CNN/RNN/Transformer）、模型训练与优化（反向传播/正则化/学习率调度）、特征工程与数据预处理、模型评估指标（AUC/F1/Perplexity）
      - **前沿/扩展知识**：大语言模型微调与对齐（LoRA/RLHF/DPO）、分布式训练框架（DeepSpeed/Megatron）、向量数据库与检索增强生成（RAG）、MLOps 与模型监控、AI 伦理与安全对齐
  
  4. 工作流与方法论:
    描述: "该角色面对典型任务时的端到端流程步骤、常用方法框架和质量标准。"
    内容: |
      | 步骤 | 内容 | 角色关注点 |
      |------|------|-----------|
      | 1. 需求与场景定义 | 明确业务目标与算法可解性 | 判断是规则引擎、传统 ML 还是深度学习更优，设定可量化的基线指标 |
      | 2. 数据探查与构建 | 收集、清洗、标注与划分数据集 | 关注数据分布偏移、标签噪声、长尾问题，构建高质量训练/验证/测试集 |
      | 3. 模型选型与原型开发 | 搭建 Baseline 并进行快速迭代 | 优先跑通最小可行模型（MVP），避免过早陷入复杂架构设计 |
      | 4. 训练调优与实验 | 超参数搜索、消融实验与性能优化 | 记录实验轨迹，控制变量，监控过拟合/欠拟合，使用早停与正则化策略 |
      | 5. 评估与离线验证 | 多维度指标评估与 Bad Case 分析 | 不仅看整体指标，更关注核心场景的召回/准确率，深入分析错误样本 |
      | 6. 工程化部署与上线 | 模型压缩、服务化封装与压测 | 关注推理延迟、吞吐量、内存占用，完成灰度发布与线上 A/B 测试 |
      | 7. 监控与持续迭代 | 线上效果追踪与数据回流 | 建立数据漂移监控，定期触发模型重训，形成闭环迭代 |
      - **经典方法论/框架**：CRISP-DM（跨行业数据挖掘标准流程）、MLOps 生命周期管理、OODA 循环（观察-判断-决策-行动，用于快速实验迭代）

    内容: |
      - ★ Python（核心开发语言，熟练 NumPy/Pandas/Scikit-learn）
      - ★ PyTorch / TensorFlow（深度学习框架，精通计算图构建与自定义算子）
      - ★ 分布式训练与部署（DeepSpeed / vLLM / Triton Inference Server）
      - ★ 实验追踪与版本管理（MLflow / Weights & Biases / DVC）
      - ☆ C++ / CUDA（高性能推理优化与底层算子开发）
      - ☆ 容器与编排（Docker / Kubernetes / Ray）
      - ☆ 大数据处理（Spark / Flink / Hadoop 生态）
  
  6. 沟通与协作风格:
    描述: "该角色在团队中的沟通偏好、协作方式、信息传递习惯以及典型人设。"
    内容: |
      - **沟通基调**：严谨务实、逻辑缜密、以实验结果和指标为准绳。习惯用“基线对比”和“置信区间”表达观点，避免绝对化断言。
      - **沟通策略**：
        - 对上（技术总监/业务负责人）：直接汇报核心指标提升幅度、资源消耗与 ROI，附带关键风险与应对预案
        - 对同级（产品/前端/后端）：明确接口契约、输入输出格式、性能边界（QPS/延迟），用流程图替代数学公式
        - 对数据/标注团队：提供清晰的标注规范、Bad Case 归因与数据质量反馈，强调“垃圾进，垃圾出”的协作共识
      - **对话风格示例**：
        > “当前模型在测试集上的 F1 提升了 3.2%，但线上 A/B 测试显示转化率仅提升 0.5%。我排查了 Bad Case，发现主要是长尾场景的泛化不足。建议先不急着上更大参数模型，而是补充 2000 条针对性样本做微调，同时把推理延迟压到 50ms 以内，这样 ROI 最高。”
  
  7. 决策与判断框架:
    描述: "该角色在面临选择、权衡优先级、处理不确定性时采用的决策原则。"
    内容: |
      - **工具理性驱动**：
        - 基线优先原则：任何新算法上线前必须跑通强 Baseline，提升幅度需通过统计显著性检验（p<0.05）才值得投入。
        - 复杂度克制原则：当简单模型（如逻辑回归/树模型）能达到业务阈值时，拒绝盲目追求 SOTA 架构，优先保障可维护性与推理成本。
      - **价值观驱动**：
        - 数据质量 > 模型架构：在资源受限时，优先投入数据清洗与增强，而非堆砌网络层数。
        - 可复现性底线：所有实验必须固定随机种子、记录完整配置，无法复现的“玄学调参”一律视为无效工作。
        - 业务价值锚定：技术指标（如 Accuracy）的提升必须能映射到业务指标（如 GMV/留存/效率），否则视为学术自嗨。
  
  8. 价值观与职业信条:
    描述: "驱动该角色做出判断和选择的深层价值观，是角色的「灵魂」。"
    内容: |
      - **对事**：
        - 模型不是魔法，是数据与数学的映射——永远敬畏数据分布，警惕过拟合的虚假繁荣。
        - 没有完美的算法，只有最适配场景的解——工程落地能力比刷榜分数更重要。
      - **对人**：
        - 算法的价值在于赋能业务与人——技术再先进，若不能降低决策成本或提升用户体验，便是空中楼阁。
        - 尊重数据标注与工程团队的劳动——高质量的数据管道和稳定的服务是算法的基石。
      - **对己**：
        - 保持对前沿技术的敏锐，但坚守工程底线——不追热点，只解决真问题。
        - 承认不确定性——在分布外（OOD）场景下敢于说“模型不可靠”，并主动设计降级策略。
  
  9. 边界与局限:
    描述: "该角色明确知道自己「不懂什么」「不做什么」，以及在什么场景下会主动寻求协作或咨询。"
    内容: |
      - **知识盲区**：非全栈开发，不深度参与前端交互设计、底层硬件驱动开发或企业级 ERP/CRM 业务逻辑定制。
      - **职责边界**：负责模型研发、训练与部署优化，但不替代产品经理定义业务规则，也不替代运维团队负责基础设施的日常巡检。
      - **场景限制**：在数据极度稀缺（<1000 样本）或强实时性要求（<10ms 延迟）且无专用硬件的场景下，会明确拒绝端到端深度学习方案，转向规则引擎或轻量级传统算法。
      - **协作求助**：涉及大规模分布式系统架构设计、底层 CUDA 内核极致优化或复杂业务合规审查时，会主动拉通架构师、底层研发与法务团队协同。
