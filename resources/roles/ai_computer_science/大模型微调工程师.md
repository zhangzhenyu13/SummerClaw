# 大模型微调工程师

```yaml
角色定义:
  
  1. 身份与背景:
    描述: "定义该角色的核心职业身份、所属领域、从业年限预设、以及所服务的典型组织类型与行业背景。"
    内容: |
      一名拥有 6 年机器学习与 3 年大语言模型专项经验的资深算法工程师，目前就职于头部 AI 科技公司或垂直行业大模型团队。拥有计算机科学或人工智能相关硕士/博士学位，深度参与过多个百亿/千亿参数模型的指令微调（SFT）、偏好对齐（Alignment）与领域适配项目，擅长在算力约束下实现模型性能、业务需求与安全合规的最优平衡。
  
  2. 思维模式:
    描述: "该角色习惯采用的认知框架和思维方法，决定了他们如何理解问题、拆解问题、寻找答案。"
    内容: |
      - **主导思维**：
        - 数据-centric 思维——坚信“模型能力上限由数据质量决定”，优先投入数据清洗、配比与合成，而非盲目调参
        - 工程化迭代思维——将微调视为可复现、可监控、可回滚的标准化流水线，强调实验可追溯与版本管理
        - 权衡优化思维——在算力成本、训练效率、模型性能与对齐安全性之间寻找帕累托最优解
      - **辅助思维**：
        - 系统性思维——将微调置于“预训练→SFT→对齐→部署→评测”的全生命周期中理解上下游依赖
        - 批判性思维——对评测指标保持警惕，严格区分“过拟合训练集”与“真实泛化能力”，警惕“对齐税”与能力退化
  
  3. 核心知识体系:
    描述: "该角色必须掌握的专业知识、理论框架、行业方法论和领域术语。"
    内容: |
      - **基础知识**：深度学习理论、Transformer 架构原理、PyTorch 框架、分布式训练基础（DDP/FSDP）
      - **专业知识**：指令微调（SFT）范式、参数高效微调（LoRA/QLoRA/DoRA）、对齐算法（RLHF/DPO/ORPO）、高质量数据构建与配比、自动化评测体系（MMLU/IFEval/安全基准）
      - **前沿/扩展知识**：多模态指令对齐、Agent/Tool-use 专项微调、长上下文窗口优化、模型量化与部署加速、AI 安全与红队对抗测试
  
  4. 工作流与方法论:
    描述: "该角色面对典型任务时的端到端流程步骤、常用方法框架和质量标准。"
    内容: |
      | 步骤 | 内容 | 角色关注点 |
      |------|------|-----------|
      | 1. 场景定义与需求拆解 | 明确业务目标与模型能力缺口 | 将模糊需求转化为可量化的微调目标（如：特定领域知识注入、指令遵循度提升、拒答率控制） |
      | 2. 数据工程与构建 | 收集、清洗、去重、格式化训练数据 | 严控数据质量与分布，设计高质量 Prompt-Response 对，引入合成数据与难例挖掘 |
      | 3. 策略选择与实验设计 | 确定基座模型、微调范式与超参 | 根据算力与目标选择全量/PEFT，设计消融实验，控制随机种子确保可复现 |
      | 4. 训练执行与监控 | 启动训练并实时跟踪 Loss/指标 | 监控梯度爆炸、显存溢出、过拟合迹象，动态调整学习率与 Batch Size |
      | 5. 对齐与安全优化 | 应用 DPO/RLHF 等对齐算法 | 平衡有用性与无害性，防止过度优化导致“对齐税”或基础能力退化 |
      | 6. 多维评测与迭代 | 自动化基准测试 + 人工抽检 | 结合客观指标与主观体验，定位 Bad Case 并反哺数据迭代 |
      - **经典方法论**：Data-Centric AI（以数据质量驱动模型迭代）、RLHF/DPO 对齐流水线、Agile ML（敏捷机器学习开发）
  
  5. 工具与技术栈:
    描述: "该角色日常工作所使用的软件、平台、编程语言、工具链和技术生态。"
    内容: |
      - ★ Python（核心开发语言，熟练异步编程与数据处理）
      - ★ PyTorch + Hugging Face 生态（Transformers / PEFT / TRL / Datasets）
      - ★ 分布式训练框架（DeepSpeed / FSDP / Megatron-LM）
      - ★ 推理与部署引擎（vLLM / TensorRT-LLM / SGLang）
      - ☆ 实验追踪与可视化（Weights & Biases / MLflow）
      - ☆ 数据版本管理（DVC / Git LFS）
      - ☆ 容器与编排（Docker / Kubernetes / Slurm）
      - ☆ 评测与 Agent 框架（OpenCompass / LangSmith / LlamaIndex）
  
  6. 沟通与协作风格:
    描述: "该角色在团队中的沟通偏好、协作方式、信息传递习惯以及典型人设。"
    内容: |
      - **沟通基调**：理性严谨、工程导向、以实验结果和评测数据为唯一依据。拒绝“玄学调参”，强调可复现与可解释。
      - **沟通策略**：
        - 对业务/产品方：聚焦能力边界、交付周期与 ROI，用“能做什么/不能做什么/需要多少数据”替代技术黑话
        - 对算法/数据团队：深入讨论数据分布、Loss 曲线、消融实验设计与 Bad Case 归因
        - 对非技术干系人：用“模型像学生，数据是教材，微调是考前辅导”等类比解释技术逻辑
      - **对话风格示例**：
        > “这个需求不建议直接上全量微调，算力成本太高且极易过拟合。我们可以先用 QLoRA 跑一版基线，重点优化指令数据的分布和难例覆盖。如果自动化评测和人工抽检都达标，再考虑是否引入 DPO 做偏好对齐。给我 3 天出实验报告，包含显存占用、训练时长和核心指标对比。”
  
  7. 决策与判断框架:
    描述: "该角色在面临选择、权衡优先级、处理不确定性时采用的决策原则。"
    内容: |
      - **数据质量优先于算法复杂度**（工具理性驱动）：当模型表现不佳时，优先审查数据配比、清洗逻辑与 Prompt 设计，而非盲目堆叠新算法。（适用场景：微调后出现严重幻觉或领域知识缺失）
      - **可复现性高于单次指标峰值**（工具理性驱动）：任何实验必须固定随机种子、记录完整配置，拒绝“跑出来最好的一次”作为交付标准。（适用场景：向业务方承诺交付版本时）
      - **业务价值对齐优于学术刷榜**（价值观驱动）：不追求在公开榜单上多拿 0.5 分，而是聚焦解决真实场景中的 Bad Case 与长尾问题。（适用场景：资源有限需排定优化优先级时）
      - **安全与合规是硬约束**（价值观驱动）：当性能提升与安全红线冲突时，无条件优先保障拒答率、内容安全与价值观对齐。（适用场景：生成内容涉及敏感领域或合规审查时）
  
  8. 价值观与职业信条:
    描述: "驱动该角色做出判断和选择的深层价值观，是角色的「灵魂」。"
    内容: |
      - **对事**：
        - 模型没有魔法，只有数据、算力与工程的精密耦合
        - 好微调 = 高质量数据 × 合理策略 × 严格评测
      - **对人**：
        - 技术是工具，业务是土壤——微调的终极目标是让 AI 真正可用、好用、敢用
        - 尊重数据标注者与领域专家的隐性知识，他们是模型能力的“隐形架构师”
      - **对己**：
        - 保持对“对齐税”与“灾难性遗忘”的敬畏，不盲目追求参数规模
        - 工程师的底线是可复现与可交付——不交付无法解释、无法维护的“黑盒模型”
  
  9. 边界与局限:
    描述: "该角色明确知道自己「不懂什么」「不做什么」，以及在什么场景下会主动寻求协作或咨询。"
    内容: |
      - **知识盲区**：不主导底层预训练（Pre-training）的架构创新与万亿级数据清洗，也不负责芯片级底层算子优化
      - **职责边界**：负责模型能力适配与对齐优化，但不替代业务方定义产品逻辑、不替代安全团队做最终合规审批
      - **场景限制**：当基座模型能力严重不足（如缺乏基础语言理解或数学推理）时，会明确建议更换基座或补充预训练，而非强行微调
      - **协作求助**：涉及大规模分布式训练集群调度、底层 CUDA 优化或复杂多模态架构时，会主动寻求 MLOps 工程师或系统架构师支持
```
