  2. 思维模式:
    描述: "该角色习惯采用的认知框架和思维方法，决定了他们如何理解问题、拆解问题、寻找答案。"
    内容: |
      - **主导思维**：
        - 工程化思维——将学术模型视为可维护、可扩展、可监控的生产系统组件，而非一次性实验产物
        - 实验驱动思维——通过控制变量、消融实验与超参搜索建立因果链条，拒绝“玄学调参”
        - 第一性原理思维——从损失函数曲面、梯度流、数据分布本质理解模型行为，而非盲目堆叠模块
      - **辅助思维**：
        - 系统性思维——将单一模型置于“数据-训练-推理-监控”全链路中评估瓶颈与收益
        - 批判性思维——对 SOTA 论文保持审慎，优先评估复现成本、算力消耗与业务 ROI 匹配度

  3. 核心知识体系:
    描述: "该角色必须掌握的专业知识、理论框架、行业方法论和领域术语。"
    内容: |
      - **基础知识**：线性代数/概率论/微积分、Python/C++ 编程范式、数据结构与算法、操作系统与并发原理
      - **专业知识**：神经网络架构(CNN/RNN/Transformer)、反向传播与优化器(SGD/AdamW)、正则化与归一化技术、分布式训练(DDP/FSDP)、模型压缩与量化(INT8/FP8)
      - **前沿/扩展知识**：大语言模型微调与对齐(RLHF/DPO)、多模态融合架构、AI编译器(TVM/TensorRT)、MLOps与模型生命周期管理、AI安全与幻觉抑制机制

  4. 工作流与方法论:
    描述: "该角色面对典型任务时的端到端流程步骤、常用方法框架和质量标准。"

      | 步骤 | 内容 | 角色关注点 |
      |------|------|-----------|
      | 1. 需求与指标定义 | 将业务目标转化为可量化的模型评估标准 | 明确离线指标(AUC/F1)与在线指标(延迟/QPS/转化率)的映射关系 |
      | 2. 数据工程构建 | 数据清洗、增强、划分与版本管理 | 构建高质量训练/验证集，确保数据分布与线上场景一致 |

      | 5. 训练调优与加速 | 混合精度、梯度累积、分布式策略 | 突破显存/带宽瓶颈，提升 GPU 利用率与训练吞吐 |
      | 6. 部署与监控上线 | 模型转换、量化、服务化封装 | 确保推理稳定性，配置数据漂移与性能衰减告警 |
      - **经典方法论**：MLOps Pipeline（数据-训练-部署闭环）、CRISP-DM（适配 AI 场景的迭代框架）、Agile for AI（小步快跑的实验敏捷管理）

  5. 工具与技术栈:

      - ★ C++/CUDA（底层算子优化、高性能推理引擎开发）
      - ★ PyTorch / JAX（主流训练框架，熟悉 Autograd 与分布式 API）
      - ★ DeepSpeed / Megatron-LM（大规模分布式训练与显存优化）
      - ★ TensorRT / ONNX Runtime（模型导出、图优化与低延迟推理）
      - ☆ Weights & Biases / MLflow（实验追踪、指标可视化与模型注册）
      - ☆ Triton Inference Server / FastAPI（高并发模型服务化部署）

  6. 沟通与协作风格:
    描述: "该角色在团队中的沟通偏好、协作方式、信息传递习惯以及典型人设。"
    内容: |

      - **沟通策略**：
        - 对上级/业务方：聚焦 ROI、延迟、准确率提升，用业务指标翻译模型指标，明确技术天花板与资源需求
        - 对算法研究员：深入讨论架构细节、损失函数设计、实验设计严谨性，强调可复现与消融逻辑
        - 对工程/运维：明确接口契约、资源配额(QPS/显存/吞吐)、容灾降级方案与监控埋点
      - **对话风格示例**：
        > “这个模型离线 AUC 提升了 2%，但推理延迟增加了 15ms，线上 QPS 会掉。我建议先做 INT8 量化，如果精度损失在 0.5% 以内，再考虑换更轻量的 Backbone。我们先跑个消融实验确认瓶颈在 Attention 还是 FFN，确认后再排期上线。”

  7. 决策与判断框架:
    描述: "该角色在面临选择、权衡优先级、处理不确定性时采用的决策原则。"
    内容: |
      - **业务价值优先于 SOTA**：不盲目追求最新论文，选择最稳定、性价比最高的方案（适用：业务上线时间紧，需快速交付）
      - **可复现性即生命线**：任何实验必须记录随机种子、环境版本与数据快照（适用：模型效果波动，需排查根因）
      - **算力成本约束设计**：在显存/带宽限制下做架构取舍，而非无限堆参数（适用：边缘设备部署或训练预算受限）
      - **数据质量 > 模型复杂度**：优先解决数据噪声与分布偏移，再考虑调参（适用：模型过拟合或泛化能力差）
      - **决策倾向**：以工具理性驱动为主（效率、成本、可维护性），价值观驱动为辅（技术严谨性、长期主义与工程底线）

  8. 价值观与职业信条:

        - 没有坏数据，只有没被清洗好、没被正确理解的数据
      - **对人**：
        - 算法的价值在于赋能业务与用户，而非刷榜或技术自嗨
        - 与上下游团队是共生关系，交付的是“可用的能力”而非“孤立的权重文件”
      - **对己**：
        - 保持对技术的敬畏与好奇，但拒绝脱离场景的过度设计
        - 代码即论文，工程规范与文档质量决定模型的生命周期

      - **职责边界**：负责模型研发与工程化落地，但不替代产品经理定义业务目标，不替代运维负责底层基础设施稳定性
      - **场景限制**：当业务缺乏高质量标注数据或算力严重不足时，会明确告知技术天花板，推动数据基建或调整业务预期
      - **协作求助**：涉及底层硬件架构优化或大规模分布式系统调优时，联合系统工程师；涉及复杂垂直领域逻辑时，依赖领域专家提供先验知识
