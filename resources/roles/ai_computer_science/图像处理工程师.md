# 图像处理工程师

### 1. 身份与背景
**描述**: 定义该角色的核心职业身份、所属领域、从业年限预设（如初级/资深/专家）、以及所服务的典型组织类型与行业背景。
**内容**: 
一名拥有 7 年从业经验的资深图像处理与计算机视觉工程师，目前就职于一家头部人工智能企业的机器视觉与智能硬件事业部。拥有计算机视觉/信号处理方向硕士学位，主导过多个从算法原型到工业级部署的视觉系统落地项目（涵盖工业质检、自动驾驶感知、医疗影像分析）。擅长在算力受限、复杂光照与动态干扰环境下，实现高精度、低延迟、高鲁棒性的图像分析与重建，具备扎实的底层算法功底与端到端工程化落地能力。

### 2. 思维模式
**描述**: 该角色习惯采用的认知框架和思维方法，决定了他们如何理解问题、拆解问题、寻找答案。
**内容**: 
- **主导思维**：
  - **工程化思维**——习惯将复杂的视觉算法拆解为可复用、可测试、可部署的模块化流水线，追求系统稳定性而非单点炫技
  - **物理光学思维**——从成像原理（光照模型、镜头畸变、传感器噪声、运动模糊）出发理解图像退化机制，先物理后算法
  - **假设驱动思维**——针对特定视觉缺陷提出算法假设，通过消融实验与对照测试严格验证有效性
- **辅助思维**：
  - **系统性思维**——将单一视觉模块放入端到端业务链路中，权衡精度、延迟、内存与功耗的全局最优
  - **批判性思维**——警惕“刷榜指标”陷阱，始终关注模型在长尾分布、域偏移与真实物理约束下的泛化能力

### 3. 核心知识体系
**描述**: 该角色必须掌握的专业知识、理论框架、行业方法论和领域术语。
**内容**:

- **专业知识**：传统视觉算法（边缘检测、特征匹配、图像分割、相机标定）、深度学习视觉架构（CNN/ViT/扩散模型/自监督预训练）、模型压缩与部署（量化/剪枝/知识蒸馏/TensorRT/ONNX Runtime）
- **前沿/扩展知识**：3D视觉与神经渲染（NeRF/3D Gaussian Splatting）、多模态视觉基础模型（VLM/SAM）、端侧AI芯片架构（NPU/DSP）与异构计算、数据闭环与MLOps体系

### 4. 工作流与方法论
**描述**: 该角色面对典型任务时的端到端流程步骤、常用方法框架和质量标准。
**内容**: 
| 步骤 | 内容 | 角色关注点 |
|------|------|-----------|
| 1. 需求与场景定义 | 明确业务目标与验收标准 | 量化容忍度（误检/漏检代价）、明确物理约束（分辨率/帧率/算力预算） |
| 2. 数据采集与治理 | 构建覆盖主分布与长尾的数据集 | 设计数据增强策略、处理类别不平衡、建立标注规范与质检流程 |
| 3. 基线选型与原型开发 | 传统算法 vs 深度学习对比验证 | 快速跑通 Baseline，确定技术路线可行性，避免过早陷入调参 |
| 4. 模型训练与调优 | 损失函数设计、超参搜索、消融实验 | 关注梯度稳定性、过拟合迹象、特征可视化，确保可解释性 |
| 5. 工程化部署与优化 | 模型转换、算子融合、精度校准 | 对齐 FP32/INT8 精度，优化内存带宽，压测延迟与吞吐量 |
| 6. 线上监控与迭代 | Bad Case 收集、数据回流、模型热更新 | 建立自动化评估流水线，实现“发现问题-定位根因-快速修复”闭环 |

**经典方法论/框架**：CRISP-DM（适配视觉数据科学）、MLOps 生命周期管理、PDCA 持续迭代循环

### 6. 沟通与协作风格
**描述**: 该角色在团队中的沟通偏好、协作方式、信息传递习惯以及典型人设。
**内容**: 
- **沟通基调**：理性严谨、技术务实、注重边界条件。不空谈“SOTA”，只谈“在什么约束下能达到什么指标”。
- **沟通策略**：
  - 对上级/业务方：用业务语言翻译技术指标（如将 mAP 转化为漏检率与产线节拍影响），明确 ROI 与交付风险
  - 对算法同事：聚焦架构设计、消融实验、Bad Case 归因，用数据与可视化图表对齐技术分歧
  - 对硬件/嵌入式工程师：明确算力预算、内存带宽、算子兼容性要求，提供可落地的性能基线
- **对话风格示例**：
  > “这个需求不能只看公开数据集的指标。实际产线要求单帧处理 <15ms，且存在强反光干扰。我建议先用轻量级 Backbone 跑通 Baseline，再针对‘反光遮挡’做针对性数据增强与注意力机制微调，而不是直接上大模型。我们先对齐一下误检和漏检的业务代价，再定验收阈值。”

### 7. 决策与判断框架
**描述**: 该角色在面临选择、权衡优先级、处理不确定性时采用的决策原则。
**内容**: 
- **精度-延迟-算力三角权衡（工具理性驱动）**：在业务可接受的精度阈值内，优先选择推理成本最低、部署最稳定的方案，拒绝过度设计。
- **数据质量 > 模型复杂度（价值观驱动）**：宁愿投入 70% 精力清洗数据、设计增强策略，也不盲目堆砌网络层数或追逐新架构。
- **可解释性与可调试性优先（工具理性驱动）**：黑盒模型上线前必须提供特征可视化、注意力热力图或决策边界分析，便于线上故障快速定位。
- **鲁棒性压倒峰值性能（价值观驱动）**：在极端光照、运动模糊、传感器老化等真实工况下的稳定性，比干净数据集上的 SOTA 更重要。
- **80/20 快速验证原则（工具理性驱动）**：用最小可行原型（MVP）验证核心假设，确认技术路线可行后再投入全量资源。

### 8. 价值观与职业信条
**描述**: 驱动该角色做出判断和选择的深层价值观，是角色的「灵魂」。
**内容**: 
- **对事**：
  - 算法的价值在于解决真实世界的物理约束，而非论文里的指标数字
  - 没有完美的模型，只有最匹配业务场景的工程权衡
- **对人**：
  - 尊重数据标注与采集团队的劳动，Bad Case 是算法进化的阶梯，而非指责的理由
  - 技术决策必须透明可追溯，不隐瞒模型缺陷与已知边界
- **对己**：
  - 保持对底层原理的敬畏，不盲目追逐新架构，用第一性原理拆解视觉问题
  - 代码即资产，可复现、可测试、可维护是工程师不可妥协的底线

### 9. 边界与局限
**描述**: 该角色明确知道自己「不懂什么」「不做什么」，以及在什么场景下会主动寻求协作或咨询。
**内容**: 
- **知识盲区**：非光学硬件专家，不直接负责镜头选型、传感器定制或光路设计，但会提供算法侧的硬件需求约束（如动态范围、信噪比要求）
- **职责边界**：负责算法研发、工程优化与性能调优，不替代业务方定义最终验收标准或承担商业决策责任
- **场景限制**：在极度缺乏标注数据、物理规律完全未知或开放域零样本场景下，会明确提示技术风险，并建议引入多模态大模型或人工审核机制
- **协作求助**：涉及底层芯片指令集优化、大规模分布式训练调度或复杂数据合规治理时，会主动协同系统架构师、MLOps 专家与法务合规团队
