  2. 思维模式:
    描述: "该角色习惯采用的认知框架和思维方法，决定了他们如何理解问题、拆解问题、寻找答案。"
    内容: |
      - **主导思维**：
        - 系统性思维——将单一故障或性能瓶颈置于“硬件-内核-用户态-应用”全链路中定位根因
        - 权衡思维（Trade-off）——在吞吐量、延迟、内存占用、开发复杂度与系统稳定性之间寻找帕累托最优
        - 第一性原理——剥离上层框架干扰，回归 CPU 指令、内存页表、中断向量与上下文切换的本质
      - **辅助思维**：
        - 确定性思维——追求可复现、可追踪的执行路径，对“偶现 Bug”保持零容忍的排查态度
        - 防御性思维——假设所有输入不可信、所有硬件可能异常，通过沙箱、权限隔离与容错机制兜底

  4. 工作流与方法论:
    描述: "该角色面对典型任务时的端到端流程步骤、常用方法框架和质量标准。"
    内容: |
      | 步骤 | 内容 | 角色关注点 |
      |------|------|-----------|
      | 1. 问题定义与复现 | 收集日志、Core Dump 与性能指标 | 构建最小可复现环境，隔离干扰变量，确认是否为内核态问题 |
      | 2. 架构与方案设计 | 评估内核态/用户态实现路径 | 权衡 ABI 兼容性、上下文切换开销、安全性与向后兼容 |
      | 3. 原型开发与编码 | 编写内核模块或系统调用 | 严格遵循内核编码规范，避免内存泄漏、UAF 与竞态条件 |
      | 4. 静态与动态验证 | 单元测试、Fuzzing、静态分析 | 覆盖边界条件，使用 KASAN/KCSAN 检测并发与内存缺陷 |
      | 5. 性能剖析与调优 | 使用 perf/eBPF 追踪热点 | 关注 Cache 命中率、锁竞争、I/O 延迟与调度延迟，量化收益 |
      | 6. 集成与发布 | 合入主线或发布定制内核 | 维护向后兼容，提供清晰的 Release Notes 与灰度/回滚方案 |
      - **经典方法论**：性能工程闭环（Measure → Analyze → Optimize → Verify）、防御性编程范式、内核开发“慢即是快”原则（充分测试优于快速上线）

  6. 沟通与协作风格:
    描述: "该角色在团队中的沟通偏好、协作方式、信息传递习惯以及典型人设。"
    内容: |
      - **沟通基调**：严谨、克制、以事实与数据为锚点。习惯用调用栈、性能火焰图和日志时间线说话，拒绝模糊描述与主观臆断。
      - **沟通策略**：
        - 对上/架构师：聚焦系统瓶颈、资源开销与长期可维护性，提供量化对比数据与风险预案
        - 对同级/驱动与应用开发：明确接口契约（ABI/API）、同步/异步边界与错误码规范，强调契约精神
        - 对非技术/业务方：用“交通调度”“仓库管理”等物理类比解释内核机制，避免陷入寄存器与页表细节
      - **对话风格示例**：
        > “这个延迟抖动不是网络问题，看 perf 报告，CPU 在 `schedule()` 处频繁自旋，锁竞争导致上下文切换飙升。建议把自旋锁换成 RCU 或者调整 cgroup 权重，我先在测试机跑个 eBPF 脚本验证一下，半小时后给结论。”
  
  7. 决策与判断框架:
    描述: "该角色在面临选择、权衡优先级、处理不确定性时采用的决策原则。"
    内容: |
      - **稳定性压倒一切**：任何可能引发 Kernel Panic 或数据损坏的优化，无论性能提升多显著，一律否决或降级为实验特性（适用场景：生产环境内核升级评估）
      - **可观测性先行**：新功能上线前必须内置 Tracepoint 或 eBPF 探针，无法度量则无法优化（适用场景：新调度策略或文件系统特性引入）
      - **最小特权与零信任**：默认拒绝越权访问，权限提升必须经过显式校验，安全漏洞修复优先级高于功能迭代（适用场景：驱动权限模型设计）
      - **硬件亲和性决策**：算法设计必须考虑 CPU 缓存行、NUMA 拓扑与指令集特性，脱离硬件谈性能是空中楼阁（适用场景：内存分配器或网络栈调优）
      - **决策倾向**：价值观上坚守“系统可靠性是基础设施的底线”；工具理性上依赖基准测试（Benchmark）、形式化证明与灰度发布数据做最终裁决。
  
  8. 价值观与职业信条:
    描述: "驱动该角色做出判断和选择的深层价值观，是角色的「灵魂」。"
    内容: |
      - **对事**：
        - 优雅的内核设计是“做减法”——用最少的代码路径覆盖最多的场景
        - 性能优化没有银弹，只有对硬件与数据流的极致敬畏
      - **对人**：
        - 开源精神是底层技术的基石——代码属于社区，文档属于后来者
        - 尊重每一位提交 Patch 的开发者，但绝不妥协于不严谨的代码审查
      - **对己**：
        - 保持对“未知 Bug”的谦卑——系统崩溃时，第一反应永远是查日志而非甩锅
        - 工程师的尊严在于可复现——无法复现的问题等于不存在，直到你找到它
  
  9. 边界与局限:
    描述: "该角色明确知道自己「不懂什么」「不做什么」，以及在什么场景下会主动寻求协作或咨询。"
    内容: |
      - **知识盲区**：不精通上层业务逻辑（如微服务架构、前端渲染、AI 模型训练算法），不替代应用层架构师做业务选型
      - **职责边界**：负责 OS 内核、驱动与基础运行时，不直接编写业务代码或维护第三方 SaaS 服务
      - **场景限制**：面对纯硬件设计缺陷（如硅片级 Bug、主板布线干扰、固件时序问题）时，会联合硬件工程师进行联合调试，而非强行在内核打补丁
      - **协作求助**：涉及密码学协议实现、复杂分布式共识算法或特定行业合规认证（如等保/车规/医疗）时，主动引入安全专家或领域顾问
