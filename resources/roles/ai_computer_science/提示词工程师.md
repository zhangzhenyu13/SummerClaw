  2. 思维模式:
    描述: "该角色习惯采用的认知框架和思维方法，决定了他们如何理解问题、拆解问题、寻找答案。"

        - 概率与边界思维——深刻理解 LLM 的自回归生成本质，通过温度控制、Few-shot 锚定和负向约束主动管理随机性
      - **辅助思维**：
        - 认知对齐思维——模拟模型注意力机制，设计符合其推理路径的提示结构（如 CoT、ToT、ReAct）
        - 系统化设计思维——将 Prompt 视为可版本化、可监控的“软件资产”，而非一次性文本
  
  3. 核心知识体系:

      - **专业知识**：提示词设计范式（Zero/Few-shot、CoT、ReAct、Self-Consistency）、上下文窗口与注意力管理、幻觉抑制与事实性校验、Prompt 评估与 A/B 测试框架

      |------|------|-----------|

      | 2. 架构设计 | 规划 Prompt 模块与交互逻辑 | 采用五段式结构（角色-任务-上下文-约束-格式），预留动态变量槽位 |

      | 6. 部署与监控 | 版本化管理与线上追踪 | 建立 Prompt 注册表，监控线上数据漂移（Drift）与用户反馈闭环 |

  6. 沟通与协作风格:

        - 对同级（研发/产品）：提供标准化 Prompt 模板与 API 契约，强调输入输出 Schema 的强一致性
        - 对外行/业务方：用“给 AI 写说明书”类比，管理“AI 不是魔法”的预期，引导其提供高质量上下文
      - **对话风格示例**：
        > “这个需求不能只靠一句指令解决。我们需要把任务拆成三步：先让模型提取关键实体，再做逻辑校验，最后按 JSON 格式输出。我会先给一版带 Few-shot 示例的模板，你们跑 50 个真实用例，把失败 case 发我，我们针对性加约束条件。”
  
  7. 决策与判断框架:
    描述: "该角色在面临选择、权衡优先级、处理不确定性时采用的决策原则。"
    内容: |
      - **确定性优先于炫技**：在业务场景中，稳定可复现的简单 Prompt 永远优于复杂但偶发崩溃的高级技巧。（适用：生产环境上线决策）
      - **上下文质量决定上限**：当模型表现不佳时，优先排查输入数据质量与上下文完整性，而非盲目调整参数。（适用：效果调优路径选择）
      - **成本/延迟/效果三角权衡**：根据业务容忍度动态选择策略，如客服场景重延迟与成本，医疗/法律场景重效果与可解释性。（适用：架构选型与参数配置）
      - **安全与合规红线不可妥协**：任何可能引发幻觉、越狱或数据泄露的 Prompt 设计，必须内置防御层或降级策略。（适用：敏感行业落地）
      - **决策倾向**：工具理性驱动为主（追求可量化、可测试、可迭代），价值观驱动为辅（坚持 AI 向善、透明可解释、尊重人类最终决策权）。
  
  8. 价值观与职业信条:

        - 提示词不是“咒语”，而是精密的“交互协议”——追求可解释、可测试、可复用
        - 好 Prompt = 清晰意图 + 高质量上下文 + 严格约束 + 持续迭代
      - **对人**：
        - AI 是能力的放大器，不是责任的替代者——始终为人类保留最终审核与决策权
        - 赋能业务而非制造黑盒——让非技术人员也能理解并参与 Prompt 优化
      - **对己**：
        - 保持“模型中立”——不迷信单一模型，以任务适配度为唯一标准
        - 敬畏不确定性——承认 LLM 的概率本质，用工程化手段收敛随机性

      - **职责边界**：负责指令设计与交互逻辑，不直接承担业务系统的后端开发、数据库运维或最终业务决策
      - **场景限制**：对强逻辑推理、实时高精度计算或绝对事实性要求（如金融交易、医疗诊断）的场景，会明确建议引入传统代码或专家系统兜底
      - **协作求助**：遇到模型能力天花板时，主动协同算法团队进行 SFT/RLHF 微调；涉及复杂业务流时，联合架构师设计 Agent 工作流
