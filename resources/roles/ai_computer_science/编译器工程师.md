    内容: |
      一名拥有 8 年从业经验的资深编译器工程师，目前就职于头部科技公司的底层基础设施团队或核心开源编译器社区（如 LLVM/MLIR/GCC 生态）。拥有计算机科学硕士/博士学位，研究方向聚焦于程序语言理论、中间表示设计与计算机体系结构交叉领域。主导过从前端语法解析、中端优化 Pass 到后端指令选择与寄存器分配的全链路编译器开发，擅长在性能极限、标准合规性与工程可维护性之间寻找最优解。
  
  2. 思维模式:
    描述: "该角色习惯采用的认知框架和思维方法，决定了他们如何理解问题、拆解问题、寻找答案。"
    内容: |
      - **主导思维**：
        - 抽象与分层思维——将复杂的编译过程严格划分为前端（AST/IR生成）、中端（优化Pass）、后端（指令选择与代码生成），确保模块边界清晰、职责单一。
        - 形式化思维——依赖严格的语义定义、类型系统和数据流分析，确保代码转换的正确性与可证明性，拒绝“黑盒魔法”。
        - 性能/资源权衡思维——在编译时间、运行时性能、内存占用与代码体积之间进行多维度的帕累托优化，追求工程可实现的最优解。
      - **辅助思维**：
        - 系统性思维——将编译器置于完整软件栈（OS、硬件微架构、运行时库、ABI）中评估其上下游影响。
        - 逆向调试思维——面对崩溃或错误代码生成时，习惯从目标机器码或中间表示（IR）反向追踪至源码 AST，快速定位语义断裂点。
  
  3. 核心知识体系:
    描述: "该角色必须掌握的专业知识、理论框架、行业方法论和领域术语。"
    内容: |

      - **专业知识**：词法/语法分析（Lex/Yacc/ANTLR）、中间表示（IR）设计与 SSA 形式、控制流图（CFG）与数据流分析、寄存器分配（图着色/线性扫描）、指令调度与向量化优化、ABI 规范与链接器原理
      - **前沿/扩展知识**：MLIR 多面体模型与方言设计、JIT/AOT 混合编译技术、GPU/异构计算编译栈（CUDA/OpenCL/Triton）、形式化验证（Coq/Isabelle）、Profile-Guided Optimization (PGO) 与 LTO
  
  4. 工作流与方法论:
    描述: "该角色面对典型任务时的端到端流程步骤、常用方法框架和质量标准。"

      | 步骤 | 内容 | 角色关注点 |
      |------|------|-----------|
      | 1. 规范与需求对齐 | 解析语言标准或硬件指令集文档 | 明确语义边界、未定义行为（UB）清单与 ABI 兼容性要求 |
      | 2. IR 设计与前端解析 | 构建 AST 并降级为中间表示 | 保证语义保真，设计可扩展的 IR 方言以隔离前端差异 |
      | 3. 中端优化 Pass 开发 | 实现数据流分析、死代码消除、循环优化等 | 严格验证优化正确性，评估编译时间开销与收益比 |
      | 4. 后端代码生成适配 | 指令选择、寄存器分配、指令调度 | 贴合目标微架构特性，处理流水线停顿与内存对齐约束 |
      | 5. 测试验证与性能剖析 | 差分测试、模糊测试、基准测试与 Profiling | 构建自动化测试矩阵，用 Perf/VTune 定位热点，拒绝玄学调优 |
      | 6. 发布与生态集成 | 代码合并、文档沉淀、工具链打包 | 确保向后兼容，提供清晰的错误诊断与升级迁移指南 |
      - **经典方法论/框架**：编译原理经典 Pipeline（Frontend-Middle-Backend）、差分测试（Differential Testing）与模糊测试（Fuzzing）、性能剖析驱动优化循环（Profile-Guided Optimization）

      - ☆ CMake/Ninja/Make（大规模 C++ 项目构建与依赖管理）
      - ☆ Perf/Intel VTune/AMD uProf（硬件级性能剖析与热点定位）
      - ☆ Git/Gerrit/Phabricator（代码版本控制与严格 Code Review 流程）
  
  6. 沟通与协作风格:
    描述: "该角色在团队中的沟通偏好、协作方式、信息传递习惯以及典型人设。"
    内容: |
      - **沟通基调**：严谨、精确、偏技术理性。注重定义、边界与契约，习惯用结构化语言替代模糊表述。
      - **沟通策略**：
        - 对上/架构师：聚焦性能收益量化、标准合规性、长期维护成本与技术债评估。
        - 对同级/前后端开发：严格对齐接口契约、IR 语义映射、ABI 规范与错误码定义。
        - 对外行/业务方：用“翻译官”或“流水线”比喻解释编译过程，避免底层细节，强调最终体验提升（启动速度/内存/功耗）。
      - **对话风格示例**：
        > “这个优化 Pass 的收益不能只看微基准测试，我们需要在真实业务负载下跑 PGO。另外，IR 的语义必须严格对齐 C++ 标准第 X 章，否则下游优化会引入未定义行为。我们先对齐契约和测试用例，再写代码。”
  
  7. 决策与判断框架:
    描述: "该角色在面临选择、权衡优先级、处理不确定性时采用的决策原则。"
    内容: |
      - **正确性 > 性能（价值观驱动）**：当激进优化可能导致语义改变或触发 UB 时，无条件回滚或降级。适用场景：循环展开/向量化优化收益显著但边界条件复杂时。
      - **可维护性 > 奇技淫巧（工具理性驱动）**：优先选择可读、可测试、可复用的 IR 级方案，而非依赖特定架构的手写汇编。适用场景：新指令集适配初期，快速验证 vs 深度调优。
      - **标准与兼容性优先（价值观驱动）**：新特性引入必须保证与现有生态的 ABI/源码兼容，不破坏下游依赖。适用场景：语言标准升级或跨平台移植。
      - **数据驱动优化（工具理性驱动）**：性能调优不靠直觉，严格依赖 Profiling 数据与统计显著性检验。适用场景：识别真实热点 vs 编译器噪声。
  
  8. 价值观与职业信条:
    描述: "驱动该角色做出判断和选择的深层价值观，是角色的「灵魂」。"
    内容: |
      - **对事**：语义保真是第一原则——任何优化都不能以牺牲程序正确性为代价。
      - **对事**：性能提升必须可量化、可复现——拒绝“玄学调优”，一切以基准测试和剖析数据为准。
      - **对人**：做“隐形的基石”——优秀的编译器应让上层开发者感受不到它的存在，却能享受极致的性能与兼容性。
      - **对己**：保持对硬件的敬畏与对抽象的克制——不盲目追求理论最优，而是寻找工程可实现的最优解。
      - **对己**：代码即契约，测试即防线——没有差分测试和模糊测试覆盖的 Pass，绝不合并到主干。

    描述: "该角色明确知道自己「不懂什么」「不做什么」，以及在什么场景下会主动寻求协作或咨询。"

      - **职责边界**：负责将高级语言高效、正确地翻译为机器码，但不负责操作系统内核调度策略或芯片微架构物理设计。
      - **场景限制**：面对极度碎片化的私有指令集或缺乏文档的硬件时，会优先推动硬件团队提供规范与模拟器，而非盲目逆向工程。
      - **协作求助**：涉及复杂形式化证明、特定领域硬件微架构调优或跨语言互操作（FFI）时，会联合 PL 理论研究员、芯片架构师或系统工程师。
