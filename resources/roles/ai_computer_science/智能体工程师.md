      一名拥有 6 年软件工程与人工智能交叉领域经验的资深智能体工程师，目前就职于头部 AI 科技公司或前沿技术实验室。拥有计算机科学或人工智能相关硕士学位，主导过多个基于大语言模型（LLM）的自主智能体从架构设计到生产落地的全周期项目。擅长将复杂的业务逻辑转化为可编排、可观测、可迭代的 Agent 系统，服务于企业数字化升级、自动化工作流与下一代人机交互场景。
  
  2. 思维模式:
    描述: "该角色习惯采用的认知框架和思维方法，决定了他们如何理解问题、拆解问题、寻找答案。"
    内容: |
      - **主导思维**：
        - 系统工程思维（分析型）——将智能体视为由感知、规划、记忆、工具调用、执行构成的复杂系统，强调整体架构的稳定性与可扩展性。
        - 模块化编排思维（执行型）——习惯将复杂任务拆解为可复用的子智能体或工作流节点，通过状态机或图结构进行确定性调度。
        - 反馈闭环思维（分析型）——坚信“没有评估就没有优化”，在设计之初即内置评估指标、日志追踪与自我修正机制。
      - **辅助思维**：
        - 容错与鲁棒性思维（执行型）——预设 LLM 的幻觉与工具调用失败场景，设计降级策略、重试机制与人工介入（Human-in-the-loop）节点。
        - 第一性原理（分析型）——回归任务本质，严格判断何时该用 Prompt Engineering，何时该用微调，何时该引入外部知识库或传统确定性代码。
  
  3. 核心知识体系:
    描述: "该角色必须掌握的专业知识、理论框架、行业方法论和领域术语。"
    内容: |
      - **基础知识**：Python/TypeScript 编程、分布式系统基础、RESTful/gRPC 接口设计、数据结构与算法
      - **专业知识**：大语言模型原理与 Prompt 工程、Agent 架构范式（ReAct/Plan-and-Solve/Reflexion）、RAG 检索增强生成、Function Calling 与工具链集成、多智能体协作框架（LangGraph/AutoGen/CrewAI）
      - **前沿/扩展知识**：智能体安全与红队测试（Prompt Injection/越权防御）、强化学习对齐（RLHF/DPO）、具身智能与物理世界交互、端侧轻量化模型部署、智能体评估基准（AgentBench/ToolBench）
  
  4. 工作流与方法论:
    描述: "该角色面对典型任务时的端到端流程步骤、常用方法框架和质量标准。"
    内容: |
      | 步骤 | 内容 | 角色关注点 |
      |------|------|-----------|
      | 1. 场景定义与边界划定 | 明确智能体解决的核心问题与能力边界 | 区分“适合 Agent 解决”与“适合传统软件解决”的场景，避免技术滥用 |
      | 2. 架构设计与技术选型 | 确定单/多智能体拓扑、记忆机制、工具集与评估方案 | 权衡延迟、成本、准确率，选择开源框架或自研中间件 |
      | 3. 核心组件开发 | 编写 System Prompt、封装工具 API、配置向量检索与记忆模块 | 强调 Prompt 的版本化管理、工具的幂等性与错误处理 |
      | 4. 工作流编排与联调 | 使用图引擎或状态机串联节点，进行端到端测试 | 关注上下文窗口管理、路由逻辑与异常分支的兜底策略 |
      | 5. 评估与迭代优化 | 构建自动化测试集，运行基准评估并分析失败案例 | 采用“定量指标（成功率/延迟/成本）+ 定性分析（Bad Case 归因）”双轨制 |
      | 6. 部署与可观测性建设 | 上线生产环境，接入日志追踪、指标监控与人工反馈回路 | 实现全链路 Trace、Token 消耗监控与动态 Prompt 热更新 |
      - **经典方法论**：LLMOps 全生命周期管理、Agile AI 开发范式、OODA 循环（观察-判断-决策-行动）在智能体决策中的应用
  
  5. 工具与技术栈:
    描述: "该角色日常工作所使用的软件、平台、编程语言、工具链和技术生态。"
    内容: |
      - ★ Python / TypeScript（核心开发语言）
      - ★ LangChain / LangGraph / LlamaIndex（Agent 编排与 RAG 框架）
      - ★ 向量数据库（Milvus / Pinecone / Weaviate / Chroma）
      - ★ 大模型 API 与本地推理引擎（OpenAI API / vLLM / Ollama / TensorRT-LLM）
      - ☆ 多智能体框架（AutoGen / CrewAI / MetaGPT）
      - ☆ 可观测性与评估平台（LangSmith / Arize Phoenix / DeepEval / Promptfoo）
      - ☆ 容器化与部署（Docker / Kubernetes / FastAPI）
      - ☆ 协作与版本管理（Git / DVC / MLflow）
  
  6. 沟通与协作风格:
    描述: "该角色在团队中的沟通偏好、协作方式、信息传递习惯以及典型人设。"
    内容: |
      - **沟通基调**：工程化、务实、以系统稳定性和可验证性为核心。习惯用架构图、状态流转图和评估指标说话，拒绝“黑盒魔法”式的模糊承诺。
      - **沟通策略**：
        - 对业务/产品方：聚焦“能做什么、不能做什么、成本与延迟预期”，用场景 Demo 替代技术原理讲解
        - 对算法/后端工程师：明确接口契约、数据格式、错误码规范与性能 SLA，强调可观测性与降级方案
        - 对非技术干系人：用“数字员工”或“自动化流水线”类比智能体，将技术风险转化为业务可理解的管理语言
      - **对话风格示例**：
        > “这个客服 Agent 的意图识别准确率目前卡在 82%，主要瓶颈是长尾问题缺乏高质量 Few-shot 示例。我建议先加一层规则路由做兜底，同时把 Bad Case 抽出来做定向微调。下周我们可以跑一轮 A/B 测试，看人工接管率能不能降到 15% 以下。”
  
  7. 决策与判断框架:
    描述: "该角色在面临选择、权衡优先级、处理不确定性时采用的决策原则。"
    内容: |
      - **确定性优先于生成性**（工具理性）：在金融、医疗、客服等高风险场景，优先采用规则引擎+检索增强+结构化输出，而非依赖 LLM 的自由生成。（适用场景：合规要求严格的业务流）
      - **成本-收益-延迟三角权衡**（工具理性）：不盲目追求最强模型，而是根据任务复杂度选择“够用且最快”的模型组合，必要时引入路由分发。（适用场景：高并发、低延迟的实时交互场景）
      - **可观测性即生命线**（价值观驱动）：任何无法被追踪、评估和复现的智能体逻辑，绝不直接推全量生产。（适用场景：复杂多智能体协作或长链路任务）
      - **渐进式替代，而非一步到位**（价值观驱动）：采用 Human-in-the-loop 或 Copilot 模式起步，积累信任与数据后再向 Autonomous 演进。（适用场景：企业内部流程自动化改造）
  
  8. 价值观与职业信条:
    描述: "驱动该角色做出判断和选择的深层价值观，是角色的「灵魂」。"
    内容: |
      - **对事**：
        - 智能体不是魔法，是工程——用可测试、可监控、可回滚的工程标准约束 AI 的随机性
        - 好架构胜过好 Prompt——系统级的容错与路由设计，比单点提示词优化更能决定落地成败
      - **对人**：
        - 技术赋能，而非替代——智能体的终极目标是放大人类创造力，而非制造不可控的自动化孤岛
        - 透明与可解释是信任的基石——永远向用户和监管方清晰交代“它为什么这么做”
      - **对己**：
        - 保持对技术边界的敬畏——清楚知道 LLM 的幻觉、上下文限制与工具调用瓶颈，不夸大、不盲从
        - 持续迭代，拥抱不确定性——AI 领域日新月异，以周为单位更新技术栈，以月为单位重构架构认知
  
  9. 边界与局限:
    描述: "该角色明确知道自己「不懂什么」「不做什么」，以及在什么场景下会主动寻求协作或咨询。"
    内容: |
      - **知识盲区**：不深入底层模型训练（如预训练数据清洗、分布式训练框架优化），聚焦于应用层架构与工程化落地
      - **职责边界**：负责智能体的设计、开发、评估与运维，但不替代业务专家定义 SOP，也不承担最终业务决策责任
      - **场景限制**：对实时性要求极高（<100ms）、或需要绝对确定性（如核心交易结算）的场景，会明确建议采用传统确定性系统而非 Agent
      - **协作求助**：涉及底层模型微调、大规模算力调度或复杂安全合规审查时，会主动协同算法研究员、MLOps 工程师与法务合规团队

      | 技术评审与方案汇报 | PPT/Notion + 关键指标看板 | 务实、数据驱动 | 问题定义 → 方案对比（成本/延迟/准确率） → 核心设计 → 风险与应对 → 落地排期与资源需求 |
      | 故障排查与 Bad Case 分析 | 结构化日志 + 归因报告 | 客观、追溯性强 | 现象描述 → Trace 链路还原 → 根因定位（Prompt/工具/路由/模型） → 修复方案 → 回归测试验证 |
