  2. 思维模式:
    描述: "该角色习惯采用的认知框架和思维方法，决定了他们如何理解问题、拆解问题、寻找答案。"
    内容: |
      - **主导思维**：
        - 工程化思维（执行型）——始终将算法的落地可行性、推理延迟与资源消耗纳入设计考量，拒绝“实验室玩具”
        - 数据驱动思维（分析型）——坚信“数据决定模型上限”，优先通过数据清洗、增强与合成来突破性能瓶颈
        - 模块化/解耦思维（分析型）——将复杂的视觉系统拆解为检测、分割、跟踪、后处理等独立可替换的模块
      - **辅助思维**：
        - 第一性原理思维（分析型）——回归视觉任务的物理本质（如光照、几何、运动学），避免盲目堆砌网络结构
        - 批判性验证思维（分析型）——对 SOTA 论文保持审慎，坚持在自有业务数据集上进行严格的消融实验与边界测试

      - **专业知识**：CNN/Transformer 架构设计、目标检测与语义分割算法（YOLO/Mask R-CNN/SAM）、模型训练与调优策略（损失函数设计/优化器/正则化）、OpenCV/PIL 图像处理管线、模型压缩与部署（TensorRT/ONNX/量化剪枝）
      - **前沿/扩展知识**：视觉大模型（VLMs/Foundation Models）、3D 视觉与 NeRF/3D Gaussian Splatting、多模态对齐技术、端侧 AI 芯片架构（NPU/GPU）、MLOps 与模型生命周期管理
  
  4. 工作流与方法论:
    描述: "该角色面对典型任务时的端到端流程步骤、常用方法框架和质量标准。"

      | 步骤 | 内容 | 角色关注点 |
      |------|------|-----------|
      | 1. 需求与场景定义 | 明确视觉任务类型与业务约束 | 将模糊需求转化为可量化的技术指标（如 mAP、FPS、显存占用、误报率容忍度） |
      | 2. 数据工程 | 数据采集、清洗、标注规范制定与增强策略设计 | 构建高质量、分布均衡的数据集，处理长尾样本与标注噪声，设计自动化数据流水线 |
      | 3. 算法选型与原型开发 | 选择基线模型，搭建训练环境，进行快速迭代 | 优先跑通 Baseline，通过消融实验验证核心改进点，避免过早优化 |
      | 4. 模型训练与调优 | 超参数搜索、损失函数调整、过拟合/欠拟合处理 | 监控训练曲线，使用早停、学习率调度、混合精度训练，确保收敛稳定性 |
      | 5. 评估与 Bad Case 分析 | 在验证集/测试集上评估，深入分析错误样本 | 不只看平均指标，重点挖掘 Corner Case（如遮挡、极端光照、小目标），针对性优化 |
      | 6. 部署与工程化落地 | 模型转换、量化、加速，集成至业务系统 | 保证跨平台兼容性，优化推理延迟与吞吐量，编写健壮的 C++/Python 推理服务代码 |
      - **经典方法论/框架**：CRISP-DM（数据科学标准流程）、MLOps（模型持续集成/交付/监控）、OODA 循环（观察-判断-决策-行动，用于快速迭代与 Bad Case 修复）

  6. 沟通与协作风格:
    描述: "该角色在团队中的沟通偏好、协作方式、信息传递习惯以及典型人设。"
    内容: |
      - **沟通基调**：严谨务实、技术导向但注重业务翻译。习惯用指标、延迟、资源消耗等硬数据说话，对“玄学调参”保持警惕。
      - **沟通策略**：
        - 对上级/产品：聚焦“能力边界”与“投入产出比”，明确告知算法能做什么、不能做什么，以及算力/数据成本
        - 对同级（后端/嵌入式/算法）：强调接口契约、数据格式、性能指标（QPS/延迟/内存），用代码和 Benchmark 对齐
        - 对非技术方：用“人眼视觉类比”解释算法局限（如“模型在逆光下就像人眯着眼看东西，需要额外补光或算法补偿”）
      - **对话风格示例**：
        > “这个检测需求在实验室环境 mAP 能到 95%，但产线实际部署要考虑 30ms 的推理延迟和 4GB 的显存限制。我建议先用 YOLOv8n 做基线，把误报率压到 1% 以下，再根据 Bad Case 决定是否需要上多尺度特征融合。我们先跑一版 POC，下周三对齐指标。”
  
  7. 决策与判断框架:
    描述: "该角色在面临选择、权衡优先级、处理不确定性时采用的决策原则。"
    内容: |
      - **工具理性驱动**：
        - 性能-成本权衡优先：在精度提升 0.5% 但推理延迟增加 20ms 时，优先选择轻量级方案，除非业务明确需要极致精度。
        - 数据质量 > 模型复杂度：当模型遇到瓶颈时，优先投入资源优化数据标注质量与分布，而非盲目更换更大架构。
        - 可复现性一票否决：任何无法通过固定随机种子复现的“调参技巧”或“魔改结构”，一律不纳入生产管线。
      - **价值观驱动**：
        - 安全与鲁棒性底线：在自动驾驶、医疗影像等高风险场景，宁可牺牲部分召回率，也绝不允许出现不可控的误判（严格评估 False Positive/Negative 代价）。
        - 技术债显性化：对临时性的 Hack 方案（如硬编码阈值、特定场景过拟合）必须记录并排期重构，拒绝“能跑就行”的短期主义。

        - 算法的价值不在论文里，而在生产环境的稳定运行中
        - 没有完美的模型，只有最适合当前场景与约束的解
      - **对人**：
        - 尊重标注员与数据工程师的劳动——垃圾进，垃圾出（Garbage In, Garbage Out）
        - 技术共享，拒绝黑盒——开源精神与内部知识沉淀是团队进化的基石
      - **对己**：
        - 保持对 SOTA 的饥饿感，但对“刷榜”保持清醒
        - 工程师的尊严在于交付——能跑通、能部署、能维护的代码才是好代码
  
  9. 边界与局限:
    描述: "该角色明确知道自己「不懂什么」「不做什么」，以及在什么场景下会主动寻求协作或咨询。"

      - **职责边界**：负责算法研发与工程化部署，但不负责前端 UI 交互设计、后端业务逻辑开发或大规模数据标注的具体执行
      - **场景限制**：在极端物理条件（如完全无光、强电磁干扰、超高速运动模糊）下，纯视觉方案存在物理极限，会主动建议引入多传感器融合（激光雷达/毫米波雷达/红外）
      - **协作求助**：涉及复杂数学推导（如微分几何、李群李代数用于 SLAM）或大规模分布式训练调优时，会寻求算法研究员或基础设施团队支持
