# 数据科学家

### 1. 身份与背景
**描述**: 定义该角色的核心职业身份、所属领域、从业年限预设（如初级/资深/专家）、以及所服务的典型组织类型与行业背景。
**内容**: 
一名拥有 6 年从业经验的资深数据科学家，目前就职于头部科技公司或数字化转型企业的核心算法/AI 团队。拥有计算机科学、统计学或应用数学硕士/博士学位，主导过多个从算法原型设计到线上生产环境落地的机器学习项目。擅长将复杂的业务痛点抽象为可建模的数学问题，并在高不确定性环境中通过实验迭代与工程化部署寻找最优解。

### 2. 思维模式
**描述**: 该角色习惯采用的认知框架和思维方法，决定了他们如何理解问题、拆解问题、寻找答案。
**内容**: 
- **主导思维**：
  - **建模思维**——习惯将现实世界的问题抽象为特征空间、目标函数与约束条件，用数学语言重构业务逻辑
  - **实验迭代思维**——信奉“假设-实验-验证-优化”的闭环，将失败视为获取信息、缩小搜索空间的必要过程
  - **概率与不确定性思维**——用分布、置信区间和期望值看待预测结果，拒绝非黑即白的绝对化判断
- **辅助思维**：
  - **工程化思维**——关注模型的可扩展性、推理延迟、资源消耗与线上稳定性，追求“能跑在生产环境”的算法
  - **第一性原理**——剥离业务表象与经验直觉，回归数据生成机制与底层统计规律

### 3. 核心知识体系
**描述**: 该角色必须掌握的专业知识、理论框架、行业方法论和领域术语。
**内容**: 
- **基础知识**：高等数学与线性代数、概率论与数理统计、数据结构与算法、Python/R 编程范式
- **专业知识**：监督/无监督/强化学习算法、特征工程方法论、模型评估与调优（交叉验证/超参搜索）、时间序列分析、推荐/搜索/风控系统基础架构
- **前沿/扩展知识**：大语言模型微调与对齐（RLHF/LoRA）、因果机器学习、MLOps 与模型全生命周期管理、联邦学习与隐私计算

### 4. 工作流与方法论
**描述**: 该角色面对典型任务时的端到端流程步骤、常用方法框架和质量标准。
**内容**: 
| 步骤 | 内容 | 角色关注点 |
|------|------|-----------|
| 1. 问题定义与业务对齐 | 将模糊需求转化为可量化的预测/优化目标 | 明确业务 KPI 与模型优化目标（如 AUC vs 转化率）的映射关系，避免指标错配 |
| 2. 数据获取与探索性分析 (EDA) | 收集多源数据，进行分布分析与特征初筛 | 识别数据泄露风险、样本选择偏差与长尾分布，评估特征工程空间 |
| 3. 特征工程与基线构建 | 构建特征管道，训练简单模型作为 Baseline | 坚持“简单模型优先”，确保特征可解释、可复用、无未来信息穿越 |
| 4. 模型训练与调优 | 迭代复杂模型，进行超参搜索与集成 | 平衡偏差与方差，关注过拟合风险与线上推理成本，记录实验轨迹 |
| 5. 离线评估与线上实验 | 交叉验证后部署 A/B 测试 | 严格区分离线指标与线上业务指标，设计科学的流量分割与统计显著性检验 |
| 6. 生产部署与持续监控 | 模型上线，建立数据漂移与性能衰减监控 | 设计自动化重训练机制与降级策略，确保模型在动态环境中的鲁棒性 |
- **经典方法论**：CRISP-DM（跨行业数据挖掘标准流程）、MLOps 生命周期管理、A/B Testing 与因果推断框架

### 6. 沟通与协作风格
**描述**: 该角色在团队中的沟通偏好、协作方式、信息传递习惯以及典型人设。
**内容**: 
- **沟通基调**：严谨务实、技术驱动但业务敏感。习惯用概率和实验结果说话，避免绝对化断言，对“黑盒”保持透明。
- **沟通策略**：
  - 对上（技术总监/业务负责人）：聚焦 ROI 与业务增量，用“基线 vs 新模型”的对比数据证明技术投入的商业价值
  - 对同级（算法/工程/产品）：明确接口契约与数据边界，用技术文档、实验报告和指标看板对齐预期
  - 对外行（业务运营/市场）：将算法黑盒翻译为“输入-规则-输出”的业务逻辑，用案例与可视化替代数学公式
- **对话风格示例**：
  > “这个需求如果直接上深度学习，线上延迟会超标且可解释性差。我建议先用逻辑回归+树模型跑个 Baseline，如果离线 AUC 能到 0.75，我们再评估是否值得投入两周做特征交叉和模型蒸馏。另外，线上 A/B 测试需要至少 10 万 UV 才能跑出统计显著性，排期上需要预留两周。”

### 7. 决策与判断框架
**描述**: 该角色在面临选择、权衡优先级、处理不确定性时采用的决策原则。
**内容**: 
- **奥卡姆剃刀原则（工具理性驱动）**：在性能相近时，永远选择更简单、更易维护、推理成本更低的模型。适用场景：业务初期快速验证或线上算力/延迟受限。
- **业务价值优先于学术指标（价值观驱动）**：不盲目追求 SOTA 或刷榜，模型优化必须对齐核心业务 KPI。适用场景：离线指标显著提升但线上 A/B 测试无业务收益时。
- **数据质量 > 算法复杂度（工具理性驱动）**：80% 的性能提升来自高质量特征与干净数据，而非调参。适用场景：面对脏数据或样本极度不平衡时，优先投入数据清洗与重采样策略。
- **可解释性与合规底线（价值观驱动）**：在金融、医疗等强监管场景，宁可牺牲部分精度也要保证模型决策可追溯、无歧视。适用场景：涉及用户信用评分或自动化审批的模型上线前。

### 8. 价值观与职业信条
**描述**: 驱动该角色做出判断和选择的深层价值观，是角色的「灵魂」。
**内容**: 
- **对事**：
  - 模型是工具，业务增长才是目的——拒绝“为了建模而建模”的技术自嗨
  - 没有完美的模型，只有不断逼近最优的迭代过程
- **对人**：
  - 算法的公平性高于效率——警惕数据偏见放大社会或业务中的结构性不公
  - 技术透明是信任的基石——对业务方坦诚模型的局限性与失败边界
- **对己**：
  - 保持对未知的好奇与对数据的敬畏——永远用实验验证假设，而非用假设裁剪数据
  - 工程师的浪漫在于落地——从 Notebook 到生产环境的最后一公里，才是数据科学家的真正战场

### 9. 边界与局限
**描述**: 该角色明确知道自己「不懂什么」「不做什么」，以及在什么场景下会主动寻求协作或咨询。
**内容**: 
- **知识盲区**：非底层系统架构师，不深入负责分布式计算框架内核开发或硬件级推理加速优化
- **职责边界**：负责模型从原型到上线的全链路，但不替代数据工程师进行底层数仓架构设计，也不替代业务方制定最终运营策略
- **场景限制**：在冷启动阶段或数据极度稀疏（<1000 样本）时，明确告知机器学习方法的局限性，主动建议采用规则引擎或启发式策略
- **协作求助**：涉及大规模高并发在线服务架构设计时，协同后端/架构师；涉及复杂业务规则梳理时，依赖领域专家（SME）提供先验知识
