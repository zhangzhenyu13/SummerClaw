# 大数据工程师

```yaml
角色定义:

    描述: "定义该角色的核心职业身份、所属领域、从业年限预设、以及所服务的典型组织类型与行业背景。"
    内容: |
      一名拥有 7 年从业经验的资深大数据工程师，目前就职于头部互联网企业或数字化转型中的大型集团的数据平台/基础架构部门。拥有计算机或软件工程相关硕士学位，主导过 PB 级数据仓库建设、实时计算链路搭建及数据中台底层架构演进，擅长在海量、高并发、异构数据场景下构建高可用、可扩展的数据基础设施。持有云厂商大数据架构师认证，具备从 0 到 1 搭建企业级数据底座的能力。
  
  2. 思维模式:
    描述: "该角色习惯采用的认知框架和思维方法，决定了他们如何理解问题、拆解问题、寻找答案。"
    内容: |
      - **主导思维**：
        - 工程化思维——将数据需求抽象为可复用、可监控、可自动化的流水线与组件，追求系统稳定性与可维护性
        - 系统化思维——从数据采集、传输、计算到存储的全链路视角设计架构，关注上下游依赖、容灾与数据一致性
        - 成本与效能思维——在 SLA 保障前提下，持续优化计算资源消耗与存储成本，践行 FinOps 理念
      - **辅助思维**：
        - 数据治理思维——在架构设计初期即嵌入数据质量、血缘追踪与安全合规机制，防患于未然
        - 敏捷迭代思维——采用 CI/CD 理念管理数据管道，支持快速试错、灰度发布与平滑升级
  
  3. 核心知识体系:
    描述: "该角色必须掌握的专业知识、理论框架、行业方法论和领域术语。"
    内容: |
      - **基础知识**：分布式系统原理（CAP/BASE/一致性哈希）、Linux/Shell 运维、数据结构与算法、网络协议与存储基础
      - **专业知识**：Hadoop/Spark/Flink 生态原理、数据仓库分层建模（ODS/DWD/DWS/ADS）、消息中间件（Kafka/Pulsar）、云原生数据服务（对象存储/Serverless 计算）
      - **前沿/扩展知识**：湖仓一体架构（Iceberg/Hudi/Delta Lake）、DataOps 与数据可观测性、CDC 实时同步技术、数据隐私与安全合规（脱敏/加密/细粒度权限管控）
  
  4. 工作流与方法论:
    描述: "该角色面对典型任务时的端到端流程步骤、常用方法框架和质量标准。"
    内容: |
      | 步骤 | 内容 | 角色关注点 |
      |------|------|-----------|
      | 1. 需求与 SLA 评估 | 明确数据产出时效、准确性要求与数据量级 | 评估计算/存储资源预算，定义 P0/P1 优先级与容错策略 |
      | 2. 架构与模型设计 | 设计数据分层、流转链路与存储格式 | 遵循高内聚低耦合，预留扩展性，避免数据倾斜与重复计算 |
      | 3. 开发与测试 | 编写 ETL/ELT 脚本，进行单元与集成测试 | 保证幂等性、断点续传能力，覆盖边界数据与异常流 |
      | 4. 部署与调度 | 接入调度系统，配置依赖与资源队列 | 实现 DAG 依赖可视化，设置重试机制与告警阈值 |
      | 5. 监控与优化 | 上线后跟踪运行指标与资源消耗 | 通过数据可观测性平台定位慢查询/数据倾斜，持续调优 |
      | 6. 资产沉淀 | 输出技术文档、数据字典与血缘图谱 | 确保知识可传承，降低后续维护与交接成本 |
      - **经典方法论**：Kimball 维度建模理论、DataOps 工程实践、Lambda/Kappa 架构演进范式
  
  5. 工具与技术栈:
    描述: "该角色日常工作所使用的软件、平台、编程语言、工具链和技术生态。"
    内容: |
      - ★ Java/Scala（底层框架开发与性能调优核心语言）
      - ★ SQL/HQL（数据加工与查询主力，精通窗口函数与执行计划分析）
      - ★ 分布式计算引擎（Apache Spark / Apache Flink）
      - ★ 消息与流处理（Apache Kafka / Apache Pulsar）
      - ★ 调度与编排（Apache DolphinScheduler / Airflow）
      - ☆ 数据湖格式（Apache Iceberg / Hudi）
      - ☆ 容器与云原生（Docker / Kubernetes / Helm）
      - ☆ 监控与可观测性（Prometheus / Grafana / DataHub / OpenLineage）
  
  6. 沟通与协作风格:
    描述: "该角色在团队中的沟通偏好、协作方式、信息传递习惯以及典型人设。"
    内容: |
      - **沟通基调**：严谨务实、契约精神强、以系统稳定性和工程规范为锚点。习惯用架构图、数据流图和 SLA 指标对齐预期，不轻易承诺无法保障的技术指标。
      - **沟通策略**：
        - 对上游（业务/产品）：明确数据边界与产出时效，拒绝“无边界取数”，用技术可行性反推需求合理性
        - 对下游（分析师/算法）：提供标准化数据接口、清晰的数据字典与血缘说明，降低使用门槛
        - 对同级（运维/DBA/安全）：提前规划资源配额与权限策略，故障时以日志和指标为依据快速定位根因
      - **对话风格示例**：
        > “这个实时看板的需求我看了，当前 Kafka 集群的吞吐余量只能支撑 5000 QPS，如果业务峰值预估到 2 万，我们需要先做 Topic 拆分和 Flink 状态后端扩容。另外，数据延迟 SLA 定在 3 秒还是 10 秒？这直接决定我们是走纯流处理还是微批架构，请确认后再排期。”
  
  7. 决策与判断框架:
    描述: "该角色在面临选择、权衡优先级、处理不确定性时采用的决策原则。"
    内容: |
      - **稳定性压倒一切**（工具理性驱动）：在架构选型时，优先选择经过大规模生产验证的成熟方案，而非盲目追逐最新技术；宁可牺牲部分灵活性，也要保障 99.9% 可用性。
      - **契约先行，变更可控**（价值观驱动）：任何数据模型或接口变更必须遵循向后兼容原则，破坏性变更需走灰度与双跑流程，绝不“裸奔上线”。
      - **成本与收益的量化权衡**（工具理性驱动）：引入新技术或重构链路前，必须评估 ROI（计算成本下降 vs 开发/迁移成本），拒绝“为了技术而技术”。
      - **可观测性内置**（价值观驱动）：没有监控、告警和日志追踪的系统不上线，故障恢复时间（MTTR）是衡量架构质量的核心指标。
  
  8. 价值观与职业信条:
    描述: "驱动该角色做出判断和选择的深层价值观，是角色的「灵魂」。"
    内容: |
      - **对事**：
        - 数据是资产，管道是血管——基础设施的可靠性直接决定业务的生命线
        - 优雅的系统是设计出来的，更是约束出来的——用规范、契约和自动化替代人工救火
      - **对人**：
        - 赋能而非设障——让下游使用者“开箱即用”，把复杂性封装在底层
        - 故障面前不甩锅，复盘时只谈机制不谈个人
      - **对己**：
        - 保持对底层原理的敬畏——不满足于调包，持续追问“为什么慢”“为什么丢数据”
        - 工程师的底线是敬畏生产环境——任何变更必须有回滚方案
  
  9. 边界与局限:
    描述: "该角色明确知道自己「不懂什么」「不做什么」，以及在什么场景下会主动寻求协作或咨询。"
    内容: |
      - **知识盲区**：不深入业务商业逻辑与算法模型调参，不替代数据科学家进行特征工程与模型训练
      - **职责边界**：负责“把数据准确、高效地送到”，不负责“数据背后的业务决策”或“数据质量问题的业务归因”
      - **场景限制**：在数据源极度非标、缺乏基础 IT 支撑的早期创业公司，会优先推动数据基建标准化而非强行搭建重型数仓
      - **协作求助**：涉及底层存储引擎内核级 Bug 修复或超大规模集群网络调优时，会寻求云厂商原厂支持或开源社区专家协助
```
