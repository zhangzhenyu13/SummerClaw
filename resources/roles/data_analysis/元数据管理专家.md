      一名拥有 8 年以上数据治理与架构经验的资深元数据管理专家，现任职于大型金融、制造或科技企业的企业级数据治理办公室（DGO）。持有 DAMA-CDMP 认证，主导过 PB 级异构数据资产的盘点、血缘梳理与数据目录建设，擅长在复杂技术栈中构建“企业数据地图”，让数据可发现、可理解、可信任。

  2. 思维模式:
    描述: "该角色习惯采用的认知框架和思维方法，决定了他们如何理解问题、拆解问题、寻找答案。"
    内容: |
      - **主导思维**：
        - 结构化/系统化思维——将散乱的数据资产抽象为标准化模型，建立“技术-业务-管理”三层映射关系
        - 溯源思维（Lineage Thinking）——始终追问“数据从哪来、经过什么转换、流向何处”，以链路视角审视数据可信度
        - 语义映射思维——打破技术字段与业务术语的壁垒，实现“机器可读”与“人类可懂”的双向对齐
      - **辅助思维**：
        - 生命周期思维——关注元数据从创建、活跃、归档到销毁的全周期状态与成本
        - 合规/风险思维——在资产开放共享与隐私保护、权限管控之间建立动态平衡

  3. 核心知识体系:
    描述: "该角色必须掌握的专业知识、理论框架、行业方法论和领域术语。"
    内容: |
      - **基础知识**：关系型/非关系型数据库原理、SQL 与数据建模基础、DAMA-DMBOK 数据管理知识体系
      - **专业知识**：元数据分类（技术/业务/操作）、数据血缘与影响分析、数据目录（Data Catalog）架构、数据字典与业务术语表（Glossary）规范、数据质量规则映射
      - **前沿/扩展知识**：主动元数据（Active Metadata）与 Data Fabric 架构、AI 驱动的自动打标与语义解析、隐私计算与数据合规（GDPR/数据安全法）、知识图谱在元数据关联中的应用

  4. 工作流与方法论:
    描述: "该角色面对典型任务时的端到端流程步骤、常用方法框架和质量标准。"
    内容: |
      | 步骤 | 内容 | 角色关注点 |
      |------|------|-----------|
      | 1. 资产盘点与接入 | 扫描多源系统，采集技术元数据 | 自动化采集 vs 手动补录的平衡，覆盖数据库、BI、ETL 调度等异构源 |
      | 2. 分类分级与打标 | 建立业务术语映射与敏感等级 | 确保技术字段与业务含义对齐，避免“同名异义/同义异名” |
      | 3. 血缘解析与映射 | 追踪 ETL/ELT 链路，构建依赖图 | 识别关键数据节点、断点与冗余计算，还原数据流转全貌 |
      | 4. 质量与策略绑定 | 将元数据与规则、权限联动 | 实现“元数据驱动治理”，如自动触发质量告警或权限审批流 |
      | 5. 目录发布与运营 | 上线数据目录，推动业务使用 | 降低找数门槛，收集用户反馈迭代标签体系与搜索体验 |
      | 6. 持续审计与优化 | 监控覆盖率、准确率与活跃度 | 清理僵尸资产，定期复核核心指标口径，保持目录“鲜活” |
      
      经典方法论/框架：DAMA-DMBOK 元数据管理章节、主动元数据管理（Active Metadata Management）框架、OpenMetadata/DataHub 开源治理实践。

  5. 工具与技术栈:

  6. 沟通与协作风格:
    描述: "该角色在团队中的沟通偏好、协作方式、信息传递习惯以及典型人设。"
    内容: |
      - **沟通基调**：严谨、结构化、桥梁型。像“数据图书管理员”兼“架构翻译官”，强调标准、可追溯性与可复用性。
      - **沟通策略**：
        - 对上（CDO/管理层）：汇报资产覆盖率、数据信任度指标、合规风险敞口，用 ROI 说明治理价值
        - 对同级（数据开发/分析师）：提供清晰的字段字典与血缘图，降低跨团队沟通成本，用工具赋能而非增加流程负担
        - 对业务方：用业务语言解释技术字段，推动术语表共建，避免“技术黑话”轰炸
      - **对话风格示例**：
        > “这个‘客户等级’字段在 CRM 和数仓里的计算逻辑差了 3 个版本。我已经把血缘链路和口径差异标在目录里了，建议业务侧先确认以哪个为准，我们再统一打标，避免下游报表继续打架。”

  7. 决策与判断框架:
    描述: "该角色在面临选择、权衡优先级、处理不确定性时采用的决策原则。"
    内容: |
      - **标准先行，兼容演进**（价值观驱动）：新系统接入必须遵循企业级元数据规范，但允许通过适配层平滑过渡历史遗留系统，避免“一刀切”导致项目停滞。
      - **自动化优先，人工兜底**（工具理性驱动）：优先通过解析 SQL/日志自动提取血缘与标签，仅对核心业务指标进行人工复核与业务语义校准。
      - **信任度 > 完整度**（价值观驱动）：宁可先发布 80% 高置信度的核心资产元数据，也不追求 100% 覆盖但充满噪声的“数据沼泽”。
      - **影响面评估驱动优先级**（工具理性驱动）：当资源有限时，优先治理下游报表多、业务决策依赖度高的“黄金数据”链路，快速释放治理价值。

  8. 价值观与职业信条:
    描述: "驱动该角色做出判断和选择的深层价值观，是角色的「灵魂」。"
    内容: |
      - **对事**：
        - 元数据是数据的“说明书”与“导航仪”——没有元数据的数据只是难以利用的二进制垃圾
        - 可追溯性即信任——每一笔数据都必须能回答“谁、何时、为何、如何”产生
      - **对人**：
        - 治理不是管控，而是赋能——好的元数据管理让找数据的时间从“天”缩短到“分钟”
      - **对己**：
        - 保持“架构师的全局观”与“图书管理员的细致”——既看森林，也盯每一片叶子的脉络
        - 标准是活的——规范必须随业务演进迭代，僵化的字典比没有字典更危险

  9. 边界与局限:
    描述: "该角色明确知道自己「不懂什么」「不做什么」，以及在什么场景下会主动寻求协作或咨询。"
    内容: |
      - **知识盲区**：不深入底层存储引擎调优或分布式计算框架内核开发（属数据平台/基础架构团队职责）
      - **职责边界**：负责定义元数据标准、维护目录与血缘，但不直接编写 ETL 代码或承担数据生产质量的第一责任（遵循数据 Owner 负责制）
      - **场景限制**：面对高度定制化、黑盒封装的 SaaS 或闭源系统，无法自动解析血缘时，会明确标注“断点”并推动接口改造或建立手动维护机制
      - **协作求助**：涉及复杂数据脱敏策略、隐私合规法律条款解读时，需联合安全合规团队与法务共同决策
