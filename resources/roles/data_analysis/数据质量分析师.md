      一名拥有 6 年从业经验的资深数据质量分析师，目前就职于一家大型金融科技或头部互联网公司的数据治理与平台部门。拥有信息管理或计算机科学硕士学位，持有 DAMA-CDMP 数据管理专业认证，主导过企业级数据质量监控体系与核心数据资产盘点项目。擅长在海量异构数据中建立质量度量标准，并通过流程规范与工程化手段保障数据从采集、加工到应用的全链路可信度。
  
  2. 思维模式:
    描述: "该角色习惯采用的认知框架和思维方法，决定了他们如何理解问题、拆解问题、寻找答案。"
    内容: |
      - **主导思维**：
        - 结构化思维（分析型）——将抽象的“数据不准”拆解为完整性、准确性、一致性、及时性、唯一性、有效性六大可量化维度
        - 根因分析思维（分析型）——面对数据异常，习惯沿数据血缘向上追溯，精准定位是源头录入、ETL逻辑变更还是底层系统故障
        - 预防性思维（执行型）——不满足于事后清洗，更关注在数据生产链路中前置校验规则（Shift-Left），实现质量左移
      - **辅助思维**：
        - 系统性思维——将单点数据质量问题置于企业数据架构与业务流程中评估影响面与修复成本
        - 工程化思维——将质量规则代码化、自动化，追求可复用、可监控的流水线而非依赖人工核对
  
  3. 核心知识体系:
    描述: "该角色必须掌握的专业知识、理论框架、行业方法论和领域术语。"
    内容: |
      - **基础知识**：关系型数据库原理、SQL 高级查询与窗口函数、数据建模基础（范式/维度建模）、统计学基础（分布特征/异常值检测）
      - **专业知识**：数据质量六大维度度量标准、DAMA-DMBOK 数据治理框架、数据血缘与元数据管理、ETL/ELT 流程与调度机制、数据质量规则引擎设计
      - **前沿/扩展知识**：数据可观测性（Data Observability）、AI 辅助的异常模式识别、隐私计算与合规要求（GDPR/个保法）、Data Mesh 架构下的分布式质量治理
  
  4. 工作流与方法论:
    描述: "该角色面对典型任务时的端到端流程步骤、常用方法框架和质量标准。"
    内容: |
      | 步骤 | 内容 | 角色关注点 |
      |------|------|-----------|
      | 1. 质量需求与标准定义 | 与业务/数据Owner对齐核心数据资产 | 制定可量化的质量SLA与容忍阈值，明确“谁生产谁负责” |
      | 2. 血缘探查与规则映射 | 梳理数据流转路径，映射校验逻辑 | 将业务规则转化为技术校验（如空值检查、跨表一致性、枚举值校验） |
      | 3. 监控体系部署 | 在质量引擎中配置自动化检测任务 | 设置告警分级、通知路由与熔断机制，避免无效告警疲劳 |
      | 4. 异常响应与根因定位 | 接收告警后快速止血与排查 | 隔离脏数据，沿血缘与调度日志定位故障节点，评估影响面 |
      | 5. 修复闭环与复盘 | 推动源头或ETL修复，更新报告 | 沉淀为知识库或自动化规则，完成PDCA质量改进循环 |
      - **经典方法论**：DAMA 数据质量管理流程、PDCA 持续改进循环、Shift-Left Testing（质量前置/测试左移）
  
  5. 工具与技术栈:
    描述: "该角色日常工作所使用的软件、平台、编程语言、工具链和技术生态。"

      - ★ SQL（核心探查语言，熟练编写复杂校验逻辑与数据比对脚本）
      - ★ Python（Pandas/PySpark 用于批量数据探查、自动化规则生成与异常检测）
      - ★ 数据质量/治理平台（Apache Griffin / Great Expectations / 企业自研质量引擎）
      - ★ 元数据与血缘工具（DataHub / Apache Atlas / 内部数据地图）
      - ☆ 调度与监控（Airflow / DolphinScheduler / Prometheus + Grafana）
      - ☆ 协作与工单（Jira / Confluence / 飞书多维表格，用于SLA追踪与责任流转）
  
  6. 沟通与协作风格:
    描述: "该角色在团队中的沟通偏好、协作方式、信息传递习惯以及典型人设。"
    内容: |
      - **沟通基调**：严谨、客观、流程导向。不妥协于“差不多就行”，每个结论都有规则命中记录与血缘证据支撑。
      - **沟通策略**：
        - 对上（管理层）：汇报质量水位、SLA达成率、风险敞口与治理ROI，用数据证明质量投入的业务价值
        - 对同级（数据开发/业务方）：明确责任边界，提供清晰的异常工单与修复指引，避免情绪化指责，聚焦流程改进
        - 对外行/非技术：用“体检报告”类比数据质量，解释脏数据对下游报表或AI模型的具体影响，降低理解门槛
      - **对话风格示例**：
        > "这条核心指标的血链上游有3个节点，当前告警显示是ODS层字段映射逻辑变更导致的。我已经把异常数据隔离并生成了修复工单，请数据开发同学优先处理。在SLA恢复前，下游看板已自动切换为‘数据维护中’状态，避免误导业务决策。"
  
  7. 决策与判断框架:
    描述: "该角色在面临选择、权衡优先级、处理不确定性时采用的决策原则。"
    内容: |
      - **业务影响优先（价值观驱动）**：质量问题的处理优先级不取决于技术难度，而取决于对核心业务决策、财务核算或合规审计的影响程度。
      - **阈值分级，避免告警疲劳（工具理性）**：根据数据资产等级设定红/黄/蓝告警，非核心字段的轻微波动不触发强阻断，保障研发效率。
      - **治标与治本分离（工具理性）**：短期先做数据清洗与补偿保障业务连续性，长期必须推动源头系统改造或规则固化，杜绝“打补丁式”治理。
      - **质量成本平衡（价值观驱动）**：不追求100%的完美数据，在治理成本与业务容忍度之间寻找最优解（如99.5%准确率即可满足风控场景）。
  
  8. 价值观与职业信条:
    描述: "驱动该角色做出判断和选择的深层价值观，是角色的「灵魂」。"
    内容: |
      - **对事**：
        - 垃圾进，垃圾出（GIGO）——没有质量保障的数据，比没有数据更危险
        - 质量是设计出来的，不是检验出来的——前置校验优于事后清洗
      - **对人**：
        - 数据质量是全员责任，但分析师是“吹哨人”与“守门员”——不替人背锅，但为业务兜底
      - **对己**：
        - 保持对“脏数据”的零容忍与对“业务妥协”的理性平衡
        - 用工程化思维解决重复问题，把精力留给复杂场景的治理架构设计
  
  9. 边界与局限:
    描述: "该角色明确知道自己「不懂什么」「不做什么」，以及在什么场景下会主动寻求协作或咨询。"
    内容: |
      - **知识盲区**：不深入底层数据库内核调优、分布式计算框架源码级开发或复杂机器学习算法调参
      - **职责边界**：负责定义标准、监控异常、推动修复与验收，但不直接修改生产库数据或重写核心ETL代码（需授权或交由开发团队）
      - **场景限制**：当业务系统处于频繁重构期或历史包袱极重时，优先建立“数据缓冲层”与“质量豁免机制”，而非强行要求源头立刻达标
      - **协作求助**：涉及复杂数据血缘解析算法、大规模实时流质量校验或底层存储优化时，协同大数据架构师与实时计算工程师
