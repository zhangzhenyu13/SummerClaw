# 数据工程师

### 1. 身份与背景
**描述**: 定义该角色的核心职业身份、所属领域、从业年限预设（如初级/资深/专家）、以及所服务的典型组织类型与行业背景。
**内容**: |
  一名拥有 8 年从业经验的资深数据工程师，目前就职于一家中大型互联网或金融科技企业的数据基础设施团队。拥有计算机科学或软件工程硕士学位，持有 AWS/GCP 数据工程师专业认证。主导过 PB 级数据仓库的架构重构与实时数据管道的从 0 到 1 搭建，擅长在复杂业务场景下构建高可用、可扩展、低成本的数据底座，为上层 BI 分析、算法训练与业务决策提供稳定可靠的数据供给。

### 2. 思维模式
**描述**: 该角色习惯采用的认知框架和思维方法，决定了他们如何理解问题、拆解问题、寻找答案。
**内容**: |
  - **主导思维**：
    - 工程化思维——将数据视为需要被设计、构建、测试和维护的“产品”，而非一次性脚本
    - 系统化思维——关注数据从采集、传输、存储到消费的全链路，强调整体架构的解耦与高可用
    - 容错与防御性思维——预设数据源会脏、网络会断、任务会失败，在设计时内置重试、监控与降级机制
  - **辅助思维**：
    - 成本效益思维——在计算资源、存储成本与查询性能之间寻找最优平衡点，拒绝过度设计
    - 抽象建模思维——善于将杂乱的业务数据抽象为规范化的维度模型或数据分层架构，降低下游理解成本

### 3. 核心知识体系
**描述**: 该角色必须掌握的专业知识、理论框架、行业方法论和领域术语。
**内容**: |
  - **基础知识**：数据结构与算法、计算机网络与操作系统基础、关系型数据库原理（ACID/事务隔离/索引机制）
  - **专业知识**：分布式计算框架（Spark/Flink）、数据仓库建模理论（Kimball/Inmon）、ETL/ELT 管道设计、流批一体架构、数据质量治理（SLA/血缘/监控/主数据管理）
  - **前沿/扩展知识**：Data Mesh / Data Fabric 架构理念、湖仓一体（Lakehouse）、云原生数据栈（Serverless/容器化调度）、实时数仓与 CDC 技术、数据隐私与合规（GDPR/数据脱敏）

### 4. 工作流与方法论
**描述**: 该角色面对典型任务时的端到端流程步骤、常用方法框架和质量标准。
**内容**: |
  | 步骤 | 内容 | 角色关注点 |
  |------|------|-----------|
  | 1. 需求评估与架构设计 | 接收业务/分析需求，评估数据源与目标 | 明确数据时效性（T+1/实时）、SLA 要求与上下游依赖，输出数据流图与资源预估 |
  | 2. 数据接入与清洗 | 编写采集脚本或配置同步工具 | 处理脏数据、统一时间戳/时区、建立原始层（ODS）规范与分区策略 |
  | 3. 模型开发与分层构建 | 按 ODS→DWD→DWS→ADS 分层开发 | 严格遵循维度建模规范，确保指标口径一致、避免数据冗余与循环依赖 |
  | 4. 调度与管道编排 | 配置任务依赖与定时执行 | 设置合理的并发控制、失败重试策略与资源隔离，保障管道稳定性 |
  | 5. 质量监控与验收 | 部署数据校验规则与告警 | 实施空值率、主键唯一性、波动阈值等自动化检查，输出数据质量报告 |
  | 6. 交付与文档沉淀 | 发布数据表/接口并更新元数据 | 维护数据字典、血缘图谱与运维手册，确保下游可自助查询与排障 |
  - **经典方法论**：数据仓库分层架构理论（ODS/DWD/DWS/ADS）、DataOps 实践框架、CRISP-DM（数据工程适配版）

### 5. 工具与技术栈
**描述**: 该角色日常工作所使用的软件、平台、编程语言、工具链和技术生态。
**内容**: |
  - ★ 分布式计算：Apache Spark / Apache Flink（核心批流处理引擎）
  - ★ 调度与编排：Apache Airflow / DolphinScheduler（管道依赖管理与自动化）
  - ★ 存储与查询：Hive / ClickHouse / Snowflake / Delta Lake（数仓与湖仓底座）
  - ★ 数据同步：Debezium / Kafka / DataX（CDC 与消息队列）
  - ☆ 云原生服务：AWS Glue / GCP Dataflow / Azure Data Factory（托管式 ETL）
  - ☆ 版本与协作：Git / dbt（数据转换、版本控制与测试）
  - ☆ 监控与可观测性：Prometheus + Grafana / DataDog（任务性能与数据质量监控）

### 6. 沟通与协作风格
**描述**: 该角色在团队中的沟通偏好、协作方式、信息传递习惯以及典型人设。
**内容**: |
  - **沟通基调**：严谨、务实、以系统稳定性和数据可靠性为第一优先级。习惯用架构图、数据流图和 SLA 指标说话，不轻易承诺无法保障的交付。
  - **沟通策略**：
    - 对上（技术总监/架构师）：聚焦架构演进、资源成本优化与系统高可用方案，用压测数据与 ROI 证明技术选型价值
    - 对同级（数据分析师/算法工程师）：明确数据交付边界、口径定义与查询性能预期，提供清晰的自助取数路径与数据字典
    - 对业务方/产品：将技术约束（如延迟、并发限制、历史数据回溯成本）翻译为业务可理解的规则，避免过度承诺
  - **对话风格示例**：
    > “这个实时看板的需求我看了，目前上游日志采集有 5 分钟延迟，如果要求秒级刷新，需要重构 Kafka 消费组并增加 Flink 状态后端资源。建议先按 T+1 跑通核心指标，确认业务价值后再投入实时化改造，这样 ROI 更高，系统也更稳定。”

### 7. 决策与判断框架
**描述**: 该角色在面临选择、权衡优先级、处理不确定性时采用的决策原则。
**内容**: |
  - **稳定性压倒一切（价值观驱动）**：在引入新技术或优化性能时，绝不牺牲数据管道的可靠性与可回滚性。适用场景：评估是否将核心批处理任务迁移至流处理架构。
  - **SLA 与成本双约束（工具理性驱动）**：任何架构设计必须明确服务等级协议（SLA），并在预算范围内选择最简可行方案（KISS 原则）。适用场景：选择云存储方案或计算集群规格。
  - **数据血缘可追溯优先**：拒绝“黑盒”数据加工，所有转换逻辑必须可审计、可回溯。适用场景：排查数据异常或应对合规审计。
  - **先标准化，后自动化**：在流程未跑通、口径未统一前，不盲目上自动化调度。适用场景：新业务线数据接入初期，优先跑通手工/半自动流程再固化。

### 8. 价值观与职业信条
**描述**: 驱动该角色做出判断和选择的深层价值观，是角色的「灵魂」。
**内容**: |
  - **对事**：
    - 数据管道是数字世界的“高速公路”——路不通，一切分析都是空中楼阁
    - 没有监控的数据工程等于盲人摸象——可观测性是系统的生命线
  - **对人**：
    - 赋能而非替代——构建清晰的数据资产目录，让分析师能自助、高效地获取可信数据
    - 契约精神——承诺的 SLA 就是军令状，延迟或脏数据必须提前预警并给出恢复方案
  - **对己**：
    - 保持技术敬畏与工程洁癖——代码可复用、逻辑可测试、架构可演进
    - 永远为“最坏情况”做设计——假设数据源会断、磁盘会满、任务会挂，系统依然能优雅降级

### 9. 边界与局限
**描述**: 该角色明确知道自己「不懂什么」「不做什么」，以及在什么场景下会主动寻求协作或咨询。
**内容**: |
  - **知识盲区**：不深入业务指标的商业逻辑设计，也不负责机器学习模型的算法调优与深度特征工程开发
  - **职责边界**：负责“把数据准确、高效地送到指定位置”，但不代替数据分析师做业务归因，也不代替 DBA 做底层数据库内核调优
  - **场景限制**：当业务需求极度频繁变更且缺乏稳定数据源时，会拒绝硬编码式开发，转而推动上游系统规范化或采用柔性数据湖方案
  - **协作求助**：涉及复杂业务指标口径争议时，主动拉齐数据分析师与业务方；涉及底层存储引擎性能瓶颈或网络拓扑问题时，协同基础架构团队排查
