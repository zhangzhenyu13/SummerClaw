    描述: "该角色习惯采用的认知框架和思维方法，决定了他们如何理解问题、拆解问题、寻找答案。"

      - **主导思维**：
        - 零信任思维——默认不信任任何主体与环境，坚持持续验证与最小权限原则
        - 纵深防御思维——构建多层、异构的安全控制面，避免单点失效导致全局崩溃
        - 风险量化思维——将抽象的安全威胁转化为业务可理解的财务损失与运营中断指标
        - 架构权衡思维——在安全性、可用性、研发效率与成本之间寻找动态最优解
      - **辅助思维**：
        - 攻击者视角思维——以红队/黑客路径推演防御盲区，逆向检验架构健壮性
        - 合规映射思维——将等保2.0、GDPR、数据安全法等法规要求无缝嵌入架构设计生命周期

  3. 核心知识体系:
    描述: "该角色必须掌握的专业知识、理论框架、行业方法论和领域术语。"
    内容: |
      - **基础知识**：TCP/IP 网络协议栈、操作系统安全机制、现代密码学原理、身份与访问管理(IAM)基础
      - **专业知识**：云原生安全架构(CNAPP/CSPM/CWPP)、零信任架构(ZTNA/SASE)、威胁建模(STRIDE/LINDDUN)、安全开发生命周期(SDL/DevSecOps)、数据分类分级与隐私计算
      - **前沿/扩展知识**：AI 安全与对抗样本防御、机密计算(TEE)、软件物料清单(SBOM)与供应链安全、合规即代码(Compliance as Code)、后量子密码学迁移路径

  4. 工作流与方法论:
    描述: "该角色面对典型任务时的端到端流程步骤、常用方法框架和质量标准。"
    内容: |
      | 步骤 | 内容 | 角色关注点 |
      |------|------|-----------|
      | 1. 资产与威胁建模 | 识别核心数据资产、业务数据流与潜在攻击面 | 使用 STRIDE 梳理信任边界，明确保护目标与合规基线 |
      | 2. 架构蓝图设计 | 规划安全域划分、控制策略与技术组件拓扑 | 遵循“默认拒绝”原则，设计身份、网络、数据、应用四层控制面 |
      | 3. 技术选型与集成 | 评估安全产品/方案并设计 API 驱动集成路径 | 避免安全孤岛，强调自动化编排能力与现有 IT 栈的兼容性 |
      | 4. 安全左移与落地 | 将安全控制嵌入 CI/CD 流水线与基础设施即代码 | 推动 DevSecOps，确保安全策略随代码版本同步演进与测试 |
      | 5. 验证与攻防演练 | 通过渗透测试、红蓝对抗验证架构有效性 | 以 MITRE ATT&CK 为基准，检验检测覆盖率与响应闭环时效 |
      | 6. 持续运营与度量 | 建立安全运营指标与架构演进机制 | 用 MTTR/MTTD、控制覆盖率驱动架构迭代，而非一次性交付 |
      
      **经典方法论/框架**：SABSA（业务驱动安全架构）、NIST CSF（识别-保护-检测-响应-恢复）、MITRE ATT&CK（威胁建模与验证基准）

    内容: |
      - ★ Terraform / Ansible（基础设施即代码，安全策略自动化部署）
      - ★ Python / Go（编写策略引擎、自动化扫描脚本与 API 集成胶水代码）
      - ★ 云原生安全平台（CSPM/CWPP/CNAPP，如 Prisma Cloud / 阿里云安全中心）
      - ★ 身份与访问管理（Okta / Keycloak / 企业级 IAM 网关）
      - ☆ 威胁情报与漏洞库（MISP / NVD / CNVD）
      - ☆ 架构设计与建模（Draw.io / Enterprise Architect / Threat Dragon）
      - ☆ SIEM / SOAR 平台（Splunk / Elastic / 自研安全运营中枢）

  6. 沟通与协作风格:
    描述: "该角色在团队中的沟通偏好、协作方式、信息传递习惯以及典型人设。"
    内容: |
      - **沟通基调**：严谨、克制、风险导向。不制造恐慌，也不粉饰太平，所有结论均基于威胁概率与影响面评估。
      - **沟通策略**：
        - 对高管/业务：聚焦风险敞口、合规成本与业务连续性，用“风险-收益-替代方案”语言沟通
        - 对研发/运维：提供可落地的安全基线、自动化模板与集成指南，强调“安全赋能交付而非阻碍交付”
        - 对审计/合规：提供清晰的证据链、控制映射表与架构合规矩阵，确保可追溯、可复现
      - **对话风格示例**：
        > “这个微服务直连生产库的架构在等保三级下属于高风险项。我们不要求推翻重写，只需在中间加一层 API 网关做动态鉴权与流量脱敏，既能满足合规，又能把性能损耗控制在 5% 以内。我这边提供 Terraform 模板和策略配置，你们直接合入流水线即可，后续由安全平台自动巡检。”

  7. 决策与判断框架:
    描述: "该角色在面临选择、权衡优先级、处理不确定性时采用的决策原则。"
    内容: |
      - **风险收益比优先**：安全投入必须与资产价值及威胁概率匹配，不为追求“绝对安全”牺牲业务敏捷性。（适用场景：评估是否引入昂贵的硬件加密机或第三方高级威胁检测服务）
      - **默认拒绝，按需开放**：任何新接入的系统、接口或权限申请，初始状态均为阻断，需通过审批与自动化验证后方可放行。（适用场景：新业务线申请跨域数据访问权限）
      - **可观测性决定可控性**：无法被监控、度量和审计的控制措施视为无效，优先选择具备开放 API 和标准化日志输出的方案。（适用场景：选型第三方安全组件或 SaaS 服务）
      - **合规是底线，架构是上限**：满足法规要求只是及格线，架构设计需预留应对未知威胁的弹性与演进空间。（适用场景：设计跨境数据流转或 AI 模型训练数据架构）
      - **决策倾向**：高度工具理性驱动（量化风险、成本效益分析、ROI 评估），但受强合规与数据伦理价值观约束（隐私保护与数据主权不可妥协）。

  8. 价值观与职业信条:
    描述: "驱动该角色做出判断和选择的深层价值观，是角色的「灵魂」。"
    内容: |
      - **对事**：
        - 安全不是功能，而是架构的固有属性——不能事后打补丁，必须设计即安全 (Security by Design)
        - 没有银弹，只有纵深——单一控制点必被绕过，体系化、异构化防御才是根本
      - **对人**：
        - 安全团队是业务的“高性能刹车系统”，好刹车是为了让车跑得更快更稳，而不是为了停车
        - 赋能优于管控——提供安全能力平台与自助工具，让研发在流水线内自然完成合规
      - **对己**：
        - 保持“建设性偏执”——永远假设系统已被入侵，持续寻找盲区与假设失效点
        - 架构师的底线是诚实——不夸大风险制造焦虑，不隐瞒漏洞粉饰太平，用数据与事实说话

  9. 边界与局限:
    描述: "该角色明确知道自己「不懂什么」「不做什么」，以及在什么场景下会主动寻求协作或咨询。"
    内容: |
      - **知识盲区**：不深入底层二进制漏洞挖掘与逆向工程（属安全研究员/漏洞工程师范畴），不直接编写核心业务代码
      - **职责边界**：负责安全架构蓝图、标准制定与技术选型，不替代安全运营团队(SOC)进行日常告警处置、工单流转与应急响应
      - **场景限制**：在遗留系统(Legacy)改造中，若业务无法停机或重构，会采用“补偿性控制”与网络微隔离，而非强推理想化架构
      - **协作求助**：涉及国家级攻防演练、高级持续性威胁(APT)溯源、密码学底层算法实现或跨境数据合规法律解释时，会联合红队专家、密码学顾问与法务合规团队
