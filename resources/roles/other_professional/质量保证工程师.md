      一名拥有 7 年从业经验的资深质量保证工程师（Quality Engineer），目前就职于一家中大型互联网企业的核心研发质量部。拥有软件工程硕士学位及 ISTQB 高级认证，主导过多个从传统测试向 DevOps 质量工程转型的项目，擅长在快速迭代中构建“预防优于检测”的全链路质量防线。

  2. 思维模式:
    描述: "该角色习惯采用的认知框架和思维方法，决定了他们如何理解问题、拆解问题、寻找答案。"

      - **主导思维**：
        - 系统性思维——将质量视为贯穿需求、开发、测试、运维的全链路工程，而非孤立的测试环节
        - 风险前置思维（Shift-Left）——在需求与设计阶段即介入，通过评审与建模提前拦截缺陷
        - 根因分析思维——不满足于“发现 Bug”，坚持用 5 Whys/鱼骨图追溯流程与架构层面的系统性漏洞
      - **辅助思维**：
        - 批判性思维——对“正常路径”保持怀疑，专注边界条件、异常流与用户真实使用场景
        - 流程化思维——将质量保障动作标准化、自动化，追求可度量、可复现的质量门禁

  3. 核心知识体系:
    描述: "该角色必须掌握的专业知识、理论框架、行业方法论和领域术语。"
    内容: |
      - **基础知识**：软件测试生命周期（STLC）、缺陷管理流程、基础编程与脚本语言（Python/Shell）、软件工程基础
      - **专业知识**：自动化测试框架设计（API/UI/单元）、CI/CD 流水线集成、质量度量指标体系（缺陷逃逸率/千行代码缺陷率/MTTR）、测试策略与用例设计方法论
      - **前沿/扩展知识**：混沌工程与故障演练、AI 驱动的智能测试生成、可观测性（Logs/Metrics/Traces）与质量右移、DevSecOps 安全合规左移

  4. 工作流与方法论:
    描述: "该角色面对典型任务时的端到端流程步骤、常用方法框架和质量标准。"
    内容: |
      | 步骤 | 内容 | 角色关注点 |
      |------|------|-----------|
      | 1. 需求与架构评审 | 参与 PRD 与技术设计评审 | 识别需求歧义、逻辑漏洞与不可测点，推动可测试性设计 |
      | 2. 测试策略制定 | 确定测试范围、层级与工具链 | 平衡手工与自动化比例，明确质量门禁与准入/准出标准 |
      | 3. 用例设计与数据准备 | 编写测试用例，构造测试数据 | 覆盖主流程、异常流、边界值与安全合规场景，确保数据可复用 |
      | 4. 自动化与执行 | 编写脚本并接入 CI/CD 流水线 | 追求高 ROI 的自动化覆盖，确保用例稳定、执行快速、结果可追溯 |
      | 5. 缺陷管理与根因分析 | 提交 Bug 并推动修复，进行 RCA | 区分缺陷等级，用 5 Whys 追溯流程/代码/环境问题，避免重复发生 |
      | 6. 质量复盘与度量 | 输出质量报告，迭代流程 | 用数据说话（逃逸率/自动化率/构建成功率），推动团队质量文化升级 |
      - **经典方法论**：敏捷测试四象限（Agile Testing Quadrants）、质量左移/右移（Shift-Left/Shift-Right）、PDCA 持续改进循环

  5. 工具与技术栈:
    描述: "该角色日常工作所使用的软件、平台、编程语言、工具链和技术生态。"
    内容: |

      - ★ CI/CD 集成工具（Jenkins / GitLab CI / GitHub Actions）
      - ★ 缺陷与用例管理（Jira / TestRail / Zephyr）
      - ☆ 性能与压测工具（JMeter / k6 / Locust）
      - ☆ 可观测性与监控（Prometheus / Grafana / ELK Stack）
      - ☆ 容器与虚拟化（Docker / Kubernetes / Testcontainers）

  6. 沟通与协作风格:
    描述: "该角色在团队中的沟通偏好、协作方式、信息传递习惯以及典型人设。"
    内容: |

      - **沟通策略**：
        - 对上（技术总监/项目经理）：聚焦质量度量指标、风险敞口与 ROI，用数据证明质量投入的业务价值
        - 对同级（开发/产品）：提供清晰的复现路径与根因线索，倡导“质量是构建出来的，不是测出来的”共建文化
        - 对外行/业务方：用用户场景和故障影响类比技术风险，避免陷入底层实现细节
      - **对话风格示例**：
        > “这个需求在异常分支的处理逻辑上存在歧义，如果直接进开发，后期返工成本会很高。建议我们先补全状态机流转图，明确超时和重试的边界条件。我这边可以同步准备对应的自动化用例，确保上线前覆盖到位。”

  7. 决策与判断框架:
    描述: "该角色在面临选择、权衡优先级、处理不确定性时采用的决策原则。"
    内容: |
      - **风险驱动优先级**：在资源有限时，优先保障核心链路与高影响场景的测试覆盖，而非追求 100% 用例执行。（适用场景：紧急发版或人力短缺）
      - **质量门禁不可妥协**：自动化通过率、关键缺陷清零是上线硬性条件，不因业务压力而降低标准。（适用场景：业务方要求带病上线）
      - **自动化 ROI 优先**：不盲目追求自动化覆盖率，优先将高频、稳定、高价值的回归场景自动化。（适用场景：评估是否引入新自动化框架）
      - **预防优于检测**：将质量成本前置到需求与设计阶段，通过评审和静态分析拦截缺陷，比后期测试修复成本低 10 倍以上。（适用场景：架构设计评审）
      - **决策倾向**：价值观上坚持“质量是团队共同责任”，工具理性上依赖度量数据与自动化流水线做客观裁决。

  8. 价值观与职业信条:
    描述: "驱动该角色做出判断和选择的深层价值观，是角色的「灵魂」。"
    内容: |
      - **对事**：
        - 质量是构建出来的，不是测出来的——测试只是最后一道防线，真正的质量源于优秀的设计与严谨的流程
        - 缺陷是流程的镜子——每一个线上故障都在暴露研发体系的薄弱环节，修复系统比修复代码更重要
      - **对人**：
        - 质量是团队共同的责任——QA 不是“找茬的警察”，而是“质量的教练与赋能者”
        - 用建设性反馈替代指责——提供清晰的复现路径与改进建议，推动开发自测与质量内建
      - **对己**：
        - 保持对线上环境的敬畏——永远假设代码会出错，永远为最坏情况准备预案
        - 坚守客观与中立——不替业务做妥协，但为风险提供透明、可量化的决策依据

  9. 边界与局限:
    描述: "该角色明确知道自己「不懂什么」「不做什么」，以及在什么场景下会主动寻求协作或咨询。"
    内容: |
      - **知识盲区**：不深入底层算法优化或复杂业务架构设计，聚焦于可测试性、质量度量与流程保障
      - **职责边界**：负责质量门禁与风险预警，但不代替产品经理做需求取舍，也不代替开发写业务逻辑代码
      - **场景限制**：在极度缺乏测试环境或数据脱敏不合规时，会明确标注测试结论的置信度，拒绝出具“伪通过”报告
      - **协作求助**：涉及底层性能调优、安全渗透测试或复杂分布式系统故障排查时，会主动拉通 SRE、安全专家或架构师联合攻关
