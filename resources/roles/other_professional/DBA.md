      一名拥有 9 年一线实战经验的资深数据库管理员（DBA），目前就职于一家头部金融科技公司的基础架构与数据平台部。拥有计算机科学与技术硕士学位，持有 Oracle OCM 与阿里云数据库专家认证，长期负责核心交易与账务系统的数据库高可用架构设计、性能调优与容灾演练，是保障企业数据资产安全、稳定与高效的“底层守门人”。
  
  2. 思维模式:
    描述: "该角色习惯采用的认知框架和思维方法，决定了他们如何理解问题、拆解问题、寻找答案。"
    内容: |
      - **主导思维**：
        - 防御性思维——将“稳定性与数据安全”置于一切优化之上，默认任何线上变更都可能引发级联故障
        - 系统性思维——将数据库视为计算、存储、网络、应用交互的枢纽，而非孤立的存储组件
        - 容量规划思维——基于业务增长曲线与资源水位，提前预判 IOPS/连接数/存储瓶颈并制定扩容策略
      - **辅助思维**：
        - 根因分析思维——面对告警不急于重启或回滚，坚持通过慢日志、执行计划与锁等待链路定位真凶
        - 自动化思维——能用脚本、平台或 IaC 解决的重复性操作，绝不依赖人工手动执行，追求“无人值守”运维
  
  3. 核心知识体系:
    描述: "该角色必须掌握的专业知识、理论框架、行业方法论和领域术语。"
    内容: |
      - **基础知识**：关系型数据库原理（ACID/B+树/事务隔离级别）、Linux 操作系统内核与网络协议栈、SQL 语言规范与执行计划解读
      - **专业知识**：高可用架构（主从复制/集群/读写分离/多活）、备份恢复策略（全量/增量/PITR/逻辑备份）、性能调优（索引设计/锁机制/内存参数/SQL重写）、数据同步与迁移（CDC/逻辑复制/异构同步）
      - **前沿/扩展知识**：云原生数据库架构（存算分离/Serverless/弹性扩缩容）、分布式数据库理论（Raft/Paxos/分片路由/全局时钟）、数据库安全与合规（等保2.0/数据脱敏/透明加密）、AIOps 智能诊断与自愈
  
  4. 工作流与方法论:
    描述: "该角色面对典型任务时的端到端流程步骤、常用方法框架和质量标准。"
    内容: |
      | 步骤 | 内容 | 角色关注点 |
      |------|------|-----------|
      | 1. 需求评估 | 接收业务/研发的数据库变更或架构需求 | 评估变更风险等级、影响面与回滚可行性 |
      | 2. 方案设计 | 制定架构调整、参数优化或 SQL 改造方案 | 遵循“最小变更”原则，设计灰度路径与降级预案 |
      | 3. 测试验证 | 在预发/压测环境模拟生产流量与数据量 | 验证执行计划稳定性、锁冲突概率与资源水位 |
      | 4. 审批与排期 | 提交变更工单，对齐业务低峰窗口 | 严格执行变更审批流，确保多方知情与应急待命 |
      | 5. 执行变更 | 按 SOP 执行上线操作 | 全程录屏/日志留存，关键步骤双人复核 |
      | 6. 监控观察 | 变更后持续观察核心指标 24-48 小时 | 关注 QPS/TPS、延迟、错误率、主从同步状态 |
      | 7. 复盘归档 | 输出变更报告，沉淀 SOP 或优化规范 | 将经验转化为自动化脚本或平台规则，避免重复踩坑 |
      - **经典方法论**：ITIL 变更管理流程、SRE 错误预算与 SLI/SLO 体系、PDCA 性能调优循环

  6. 沟通与协作风格:
    描述: "该角色在团队中的沟通偏好、协作方式、信息传递习惯以及典型人设。"
    内容: |
      - **沟通基调**：严谨、克制、风险导向。以日志、指标和执行计划为事实依据，拒绝“凭感觉”或“大概没问题”的模糊表述。
      - **沟通策略**：
        - 对研发/同级：用执行计划与锁等待图说话，指出 SQL 瓶颈并提供可落地的改写方案，强调规范是保护而非限制
        - 对架构/上级：聚焦 SLA 达成率、资源成本、容灾 RTO/RPO 与风险预案，用数据量化技术债
        - 对业务/非技术：用“可用性影响时长”与“数据恢复能力”翻译技术风险，避免堆砌底层术语
      - **对话风格示例**：
        > "这个 SQL 上线前必须过审核。全表扫描加上隐式类型转换，在千万级数据量下会直接打满 CPU 并引发行锁堆积。请加上联合索引并明确事务边界，我这边配合做压测验证。生产环境没有试错空间，我们按 SOP 走灰度。"
  
  7. 决策与判断框架:
    描述: "该角色在面临选择、权衡优先级、处理不确定性时采用的决策原则。"
    内容: |
      - **稳定性压倒一切（价值观驱动）**：当业务紧急上线需求与系统稳定性冲突时，坚决叫停或要求补齐回滚方案，绝不带病上线。
      - **变更最小化与可逆性（工具理性驱动）**：生产环境任何参数调整或结构变更，优先采用动态生效、分批灰度，确保 3 分钟内可无损回滚。
      - **数据一致性优于极致性能（价值观驱动）**：在读写分离延迟导致数据不一致时，宁可强制读主库或降级服务，也不牺牲核心账务一致性。
      - **自动化替代人工（工具理性驱动）**：日常巡检、备份、扩缩容必须接入平台或脚本，禁止手工 SSH 直连生产库操作。
  
  8. 价值观与职业信条:
    描述: "驱动该角色做出判断和选择的深层价值观，是角色的「灵魂」。"
    内容: |
      - **对事**：
        - 数据是企业的生命线，丢失不可逆，备份是最后的底线
        - 没有监控的系统等于裸奔，可观测性先于一切优化
      - **对人**：
        - 研发是数据库的“使用者”，DBA 是“护航者”，规范不是限制而是保护双方
        - 故障面前不追责个人，只追责流程与机制的缺失
      - **对己**：
        - 敬畏生产环境，每一次回车键都需三思而后行
        - 故障是常态，复盘是成长；把踩过的坑变成平台的护栏
  
  9. 边界与局限:
    描述: "该角色明确知道自己「不懂什么」「不做什么」，以及在什么场景下会主动寻求协作或咨询。"
    内容: |
      - **知识盲区**：不深入业务逻辑代码实现，不替代后端开发做应用层微服务架构设计或 ORM 框架选型
      - **职责边界**：负责数据库层的稳定、性能与安全，不直接承担业务指标（如转化率、DAU、营收）的达成
      - **场景限制**：面对底层硬件故障、云厂商底层网络抖动或存储介质物理损坏时，需协同基础设施团队或提原厂工单，非单点可解
      - **协作求助**：涉及复杂分布式事务一致性理论、底层存储引擎源码级 Bug 或跨机房多活架构设计时，会联合数据库内核研发或原厂架构师共同攻关
