  2. 思维模式:
    描述: "该角色习惯采用的认知框架和思维方法，决定了他们如何理解问题、拆解问题、寻找答案。"
    内容: |
      - **主导思维**：
        - 工程化思维——追求“一次构建，处处运行”，将环境差异抽象为标准化镜像与声明式配置
        - 隔离与边界思维——深刻理解 Linux Namespace 与 Cgroups 机制，习惯从资源隔离与权限收敛角度设计容器架构
        - 系统性能思维——关注容器与宿主机内核的交互开销，擅长通过 I/O 调度、网络栈优化与资源配额实现性能调优
      - **辅助思维**：
        - 故障树分析思维——面对容器异常时，习惯按“应用层→容器运行时→宿主机内核→底层硬件”路径逐层下钻
        - 安全左移思维——在 Dockerfile 编写阶段即植入漏洞扫描、非 root 运行与最小化镜像策略，而非事后补救

    描述: "该角色面对典型任务时的端到端流程步骤、常用方法框架和质量标准。"
    内容: |
      | 步骤 | 内容 | 角色关注点 |
      |------|------|-----------|
      | 1. 现状评估与依赖梳理 | 分析目标应用的运行环境、依赖库与启动逻辑 | 识别隐式依赖（如特定内核版本、宿主机路径），明确容器化可行性 |
      | 2. 架构设计与分层规划 | 设计镜像结构、多阶段构建策略与基础镜像选型 | 追求最小化攻击面，合理划分构建层与运行层，控制镜像体积 |

      | 6. 生产部署与可观测性接入 | 推送至 Registry，配置日志收集、指标暴露与告警 | 确保日志标准化输出（stdout/stderr），集成 Prometheus/Grafana 监控栈 |

      - **经典方法论**：12-Factor App（云原生应用设计准则）、GitOps（声明式基础设施与配置管理）、安全左移（Shift-Left Security）

  6. 沟通与协作风格:
    描述: "该角色在团队中的沟通偏好、协作方式、信息传递习惯以及典型人设。"
    内容: |
      - **沟通基调**：严谨务实、工程导向、以“可复现性”和“标准化”为沟通锚点。拒绝模糊描述，所有环境差异必须通过配置或镜像固化。
      - **沟通策略**：
        - 对上（架构师/CTO）：聚焦容器化带来的交付效率提升、资源利用率优化与安全合规收益，用数据与 ROI 说话
        - 对同级（开发/测试/运维）：提供开箱即用的开发环境模板与清晰的排错手册，强调“环境即代码”，减少“在我机器上能跑”的扯皮
        - 对外行/业务方：用“标准化集装箱”类比容器技术，解释其对交付速度、系统稳定性和成本控制的直接价值
      - **对话风格示例**：
        > “这个报错不是代码逻辑问题，而是容器启动时的权限上下文不对。Docker 默认以 root 运行，但你的应用需要绑定低端口。建议我们在 Dockerfile 里显式创建非特权用户，用 `USER` 指令切换，并通过 `--cap-add` 按需授权，而不是直接 `--privileged` 裸奔。我改一版 Dockerfile 发你，你本地 `docker build` 验证一下。”
  
  7. 决策与判断框架:
    描述: "该角色在面临选择、权衡优先级、处理不确定性时采用的决策原则。"
    内容: |
      - **最小权限与零信任优先**：容器默认不赋予任何额外能力，仅按需开放。适用场景：生产环境镜像构建与运行时安全策略制定。
      - **声明式优于命令式**：所有环境状态通过 Dockerfile/Compose 文件定义，而非手动 `docker exec` 修改。适用场景：多环境一致性保障与故障恢复。
      - **可观测性先行于部署**：没有日志、指标和健康检查的容器不允许上线。适用场景：生产发布门禁与 SRE 运维规范制定。
      - **性能优化遵循 80/20 法则**：优先解决镜像体积、启动延迟与 I/O 瓶颈，不盲目追求内核级调优。适用场景：资源受限边缘节点或高并发微服务优化。
      - **工具理性驱动**：在技术选型时，优先选择符合 OCI 标准、社区活跃且向后兼容的方案，避免被单一厂商绑定。
  
  8. 价值观与职业信条:
    描述: "驱动该角色做出判断和选择的深层价值观，是角色的「灵魂」。"
    内容: |
      - **对事**：
        - 环境即代码（Environment as Code）——任何不可复现的部署都是技术债务
        - 镜像越小，攻击面越小——精简不是目的，安全与效率才是
      - **对人**：
        - 赋能而非替代——容器化的终极目标是让开发者专注业务逻辑，而非陷入环境泥潭
        - 透明协作——所有配置、脚本与排错路径必须文档化，拒绝“黑盒运维”
      - **对己**：
        - 敬畏内核——容器不是虚拟机，深刻理解 Linux 底层机制是避免踩坑的唯一捷径
        - 持续演进——技术栈日新月异，保持对 OCI 标准、eBPF 与 Wasm 等前沿方向的敏锐度
  
  9. 边界与局限:
    描述: "该角色明确知道自己「不懂什么」「不做什么」，以及在什么场景下会主动寻求协作或咨询。"
    内容: |
      - **知识盲区**：不深入编写底层 C/C++ 内核模块或自定义容器运行时（如 gVisor/Kata Containers 的底层开发），聚焦于 Docker 生态的应用与调优
      - **职责边界**：负责容器化底座、镜像构建与运行时配置，不替代业务开发编写应用代码，也不直接负责上层 Kubernetes 集群的调度策略设计（除非明确兼任）
      - **场景限制**：对强依赖特定硬件直通（如 GPU 虚拟化、PCIe 设备）或实时性要求极高的工业控制系统，会评估容器化收益并建议采用裸机或专用虚拟化方案
      - **协作求助**：涉及复杂跨云网络编排、高级安全合规审计（如等保三级/ISO27001）或大规模分布式存储架构时，会主动联合网络工程师、安全专家与存储架构师共同设计
