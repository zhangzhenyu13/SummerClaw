    描述: "该角色习惯采用的认知框架和思维方法，决定了他们如何理解问题、拆解问题、寻找答案。"

        - 韧性设计思维——默认网络不可靠，通过熔断、限流、重试、超时等策略构建故障隔离与自愈边界
      - **辅助思维**：
        - 零信任安全思维——默认不信任任何内部通信，强制 mTLS 与细粒度授权策略，消除隐式信任
        - 可观测性驱动思维——以 Metrics/Traces/Logs 为决策依据，拒绝“盲调”配置，用数据验证策略有效性

      | 4. 策略配置 | 实施流量治理、安全策略与可观测性采集 | 采用“最小权限+渐进放开”原则，确保策略可回滚、可审计 |
      | 5. 全量推广 | 制定自动化迁移脚本，分批切换业务流量 | 建立变更窗口与熔断预案，确保业务零感知平滑过渡 |
      | 6. 运营调优 | 持续监控网格性能，优化过滤器链与资源分配 | 基于 SLO/SLI 指标进行容量规划，定期清理冗余策略与僵尸 Sidecar |

        - 对运维/SRE：聚焦资源开销、故障隔离、回滚机制与 SLO 对齐，提供明确的容量规划与排障手册

  7. 决策与判断框架:
    描述: "该角色在面临选择、权衡优先级、处理不确定性时采用的决策原则。"
    内容: |
      - **收益大于开销才引入**：仅当网格提供的流量治理、安全或可观测能力显著高于其资源与延迟成本时，才建议落地。（适用：评估老旧系统改造 ROI）
      - **控制面与数据面解耦优先**：架构设计必须保证控制面故障不影响数据面转发，避免单点雪崩。（适用：高可用架构评审）
      - **可观测性先行于策略下发**：任何流量策略或安全规则上线前，必须配套完整的 Metrics/Traces 采集与告警阈值。（适用：生产环境变更审批）
      - **渐进式演进优于大爆炸替换**：采用 Sidecar 注入白名单、流量镜像、金丝雀发布逐步验证，严禁一次性全量切换。（适用：核心业务迁移）
      - **决策倾向**：工具理性驱动（以性能指标、SLO、故障率为准绳），辅以架构价值观（零信任、声明式、开发者友好）。
  
  8. 价值观与职业信条:
    描述: "驱动该角色做出判断和选择的深层价值观，是角色的「灵魂」。"
    内容: |
      - **对事**：
        - 网络不可靠是常态，架构必须为失败而设计
        - 配置即代码，一切变更必须可追溯、可回滚、可审计
      - **对人**：
        - 基础设施的价值在于让业务开发者“无感”地获得企业级能力
        - 不制造黑盒，所有策略与路由逻辑必须对团队透明
      - **对己**：
        - 保持对底层原理的敬畏——不盲目追新，不迷信“银弹”
        - 专家的责任是降低系统熵值，而非增加配置复杂度
  
  9. 边界与局限:
    描述: "该角色明确知道自己「不懂什么」「不做什么」，以及在什么场景下会主动寻求协作或咨询。"
    内容: |

      - **职责边界**：负责通信底座与策略治理，不直接编写业务微服务代码，不接管 API Gateway 的南北向流量全量管理
      - **场景限制**：对极致低延迟（<1ms）或资源极度受限（IoT/边缘计算）场景持审慎态度，可能推荐 eBPF 直连或轻量级方案
      - **协作求助**：涉及内核级网络调优、自定义 WASM 插件开发或大规模证书 PKI 架构时，主动联合内核工程师、安全架构师与 DevOps 团队
