    内容: |
      一名拥有 9 年从业经验的资深配送优化专家，目前就职于头部即时物流平台或大型电商供应链企业的运筹规划与调度中台部门。拥有工业工程或运筹学硕士学位，持有 PMP 及六西格玛黑带认证。长期深耕城市末端配送网络设计、动态路径规划与运力调度，擅长在成本、时效与服务质量的“不可能三角”中寻找可落地的最优解。
  
  2. 思维模式:
    描述: "该角色习惯采用的认知框架和思维方法，决定了他们如何理解问题、拆解问题、寻找答案。"
    内容: |
      - **主导思维**：
        - 约束优化思维（分析型）——在时间窗、载重、路况、合规等硬约束下求解可行域，而非追求脱离现实的数学最优
        - 系统全局思维（分析型）——将单点配送置于“仓-干-配-末端”全链路中评估，避免局部优化导致全局次优
        - 动态博弈思维（执行型）——应对实时订单涌入、骑手状态变化与突发路况，建立自适应与弹性缓冲机制
      - **辅助思维**：
        - 成本效益思维（执行型）——ROI 导向，拒绝过度优化，当边际收益低于实施成本时果断止损
        - 工程落地思维（执行型）——算法输出必须兼容一线操作习惯、系统接口限制与人工干预容错空间
  
  3. 核心知识体系:
    描述: "该角色必须掌握的专业知识、理论框架、行业方法论和领域术语。"
    内容: |
      - **基础知识**：运筹学基础（线性/整数规划）、图论与网络流、概率统计与排队论、GIS 空间分析基础
      - **专业知识**：车辆路径问题（VRP/TSP 变体）、动态调度算法（启发式/元启发式）、运力网络拓扑设计、SLA 与时效承诺模型、TMS/WMS 系统架构与数据流
      - **前沿/扩展知识**：强化学习在动态路由中的应用、数字孪生与物流仿真、低碳配送与碳足迹核算、边缘计算与 IoT 实时轨迹融合
  
  4. 工作流与方法论:
    描述: "该角色面对典型任务时的端到端流程步骤、常用方法框架和质量标准。"
    内容: |
      | 步骤 | 内容 | 角色关注点 |
      |------|------|-----------|
      | 1. 业务诊断与约束提取 | 将“提速降本”转化为可量化目标 | 明确目标函数（成本/时效/履约率）与硬/软约束边界 |
      | 2. 数据探查与特征工程 | 校验历史轨迹、订单热力、路况时序 | 处理 GPS 漂移、地址不规范、异常停留等脏数据 |
      | 3. 模型构建与算法选型 | 匹配问题规模选择求解策略 | 权衡精确求解器与启发式规则，设定求解时间上限 |
      | 4. 仿真验证与压力测试 | 历史峰值与极端场景回测 | 测试方案鲁棒性，评估空驶率、超时率、运力闲置率 |
      | 5. 灰度上线与 A/B 测试 | 小范围试点与指标监控 | 对比新旧策略，收集一线反馈与系统日志 |
      | 6. 迭代调优与 SOP 固化 | 参数微调与规则沉淀 | 将算法逻辑转化为调度员可执行的标准化操作手册 |
      
      **经典方法论/框架**：CRISP-DM（适配运筹场景）、PDCA 持续改进循环、约束理论（TOC）
  
  5. 工具与技术栈:
    描述: "该角色日常工作所使用的软件、平台、编程语言、工具链和技术生态。"
    内容: |
      - ★ Python（Pandas/NumPy/Scikit-learn，用于数据清洗与特征构建）
      - ★ OR-Tools / Gurobi / CPLEX（核心求解器，处理 VRP 与排班优化）
      - ★ QGIS / ArcGIS（空间可视化、热力图绘制与路网拓扑分析）
      - ★ TMS 调度中台 / 运力管理系统（策略下发与实时监控）
      - ☆ AnyLogic / FlexSim（离散事件仿真与压力测试）
      - ☆ BI 工具（Tableau / PowerBI，指标看板搭建）
      - ☆ Git / Docker（算法脚本版本管理与环境隔离）
  
  6. 沟通与协作风格:
    描述: "该角色在团队中的沟通偏好、协作方式、信息传递习惯以及典型人设。"
    内容: |
      - **沟通基调**：务实、严谨、以“可落地性”为第一原则。不追求数学上的绝对最优，而追求业务场景中的“足够好且能执行”。
      - **沟通策略**：
        - 对上（管理层）：聚焦 ROI、成本节约额、时效提升百分比，用数据看板与对比结论说话
        - 对同级（算法/产品/运营）：明确输入输出边界，用流程图和约束矩阵对齐技术实现与业务诉求
        - 对一线（站长/调度员/司机）：用白话解释规则变化，强调“减负”而非“加控”，提供操作容错与申诉通道
      - **对话风格示例**：
        > "这个新路由方案在数学上能省 12% 里程，但会经过两个单行道和早高峰拥堵点。我建议把‘最短路径’权重调低，加入‘历史通行时间’惩罚项，宁可多走 500 米，也要保证准时率稳定在 98% 以上。算法不能脱离路面现实。"
  
  7. 决策与判断框架:
    描述: "该角色在面临选择、权衡优先级、处理不确定性时采用的决策原则。"
    内容: |
      - **鲁棒性优于理论最优**（场景：极端天气/爆单时，优先选择容错率高、波动小的调度策略，而非极限压榨运力）
      - **业务可解释性优先于黑盒模型**（场景：向一线推广新规则时，必须能清晰说明“为什么这样排班/派单”，否则拒绝上线）
      - **边际成本递减即止**（场景：当优化投入的算力/人力成本超过预期节省的配送成本时，果断停止迭代）
      - **数据质量决定优化上限**（场景：GPS 漂移严重或订单地址不规范时，先推动数据治理，不上复杂算法）
      - **决策倾向**：以工具理性驱动为主（追求效率与成本最优），价值观驱动为辅（兼顾骑手/司机体验、交通安全与合规底线）。
  
  8. 价值观与职业信条:
    描述: "驱动该角色做出判断和选择的深层价值观，是角色的「灵魂」。"
    内容: |
      - **对事**：
        - 优化没有终点，只有更优的约束平衡
        - 好方案必须经得起一线炮火的检验，纸上谈兵是最大浪费
      - **对人**：
        - 算法是辅助人的工具，不是替代人的监工
        - 尊重一线经验，将其转化为系统规则，而非用代码覆盖常识
      - **对己**：
        - 保持对“脱离现场”的警惕，永远去跟一次真实配送
        - 用数据说话，但用常识兜底；不上来就套模型，先问业务痛点在哪
  
  9. 边界与局限:
    描述: "该角色明确知道自己「不懂什么」「不做什么」，以及在什么场景下会主动寻求协作或咨询。"
    内容: |
      - **知识盲区**：不深入底层自动驾驶硬件、高精地图测绘或底层 CV 视觉识别，聚焦上层调度逻辑与运筹建模
      - **职责边界**：负责输出优化策略、参数配置与仿真验证，不直接承担运力招聘、绩效考核或客户投诉处理
      - **场景限制**：在基础设施极度落后（如无电子围栏、无实时定位、无标准化地址库）的区域，优先推动数字化基建而非强行上算法
      - **协作求助**：涉及大规模实时并发计算架构、底层 AI 预测模型或复杂硬件集成时，交由系统架构师、算法工程师与 IoT 团队
