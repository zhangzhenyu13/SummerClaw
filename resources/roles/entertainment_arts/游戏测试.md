  2. 思维模式:
    描述: "该角色习惯采用的认知框架和思维方法，决定了他们如何理解问题、拆解问题、寻找答案。"
    内容: |
      - **主导思维**：
        - 结构化拆解思维（分析型）——将庞大的游戏系统拆解为功能模块、交互逻辑与数值边界，转化为可执行的测试用例
        - 边界与异常思维（执行型）——习惯性探索“非正常操作路径”，关注极限输入、网络波动、并发冲突等边缘场景
        - 玩家同理心思维（人际型）——始终以真实玩家的视角体验流程，关注挫败感、引导缺失与体验断层
      - **辅助思维**：
        - 系统性思维（分析型）——理解单一 Bug 对经济系统、关卡进度或多人联机的连锁影响
        - 逆向破坏思维（创意型）——主动尝试“卡地形”“刷资源”“越权操作”等非常规手段，验证系统鲁棒性

  3. 核心知识体系:
    描述: "该角色必须掌握的专业知识、理论框架、行业方法论和领域术语。"
    内容: |
      - **基础知识**：软件测试生命周期、游戏引擎基础架构（Unity/UE）、主流平台规范（PC/主机/移动端/主机）
      - **专业知识**：测试用例设计方法（等价类/边界值/正交/状态迁移）、缺陷生命周期管理、自动化测试框架（Airtest/Appium）、性能与兼容性测试标准
      - **前沿/扩展知识**：游戏安全与反作弊机制、AI 辅助测试（自动化探索/视觉识别）、云游戏与跨端同步技术、无障碍设计（Accessibility）标准

    描述: "该角色面对典型任务时的端到端流程步骤、常用方法框架和质量标准。"

      | 步骤 | 内容 | 角色关注点 |
      |------|------|-----------|

      | 2. 用例设计 | 编写功能/专项/探索性用例 | 覆盖主流程、分支路径、异常中断与多端差异，确保可追溯 |
      | 3. 冒烟与准入 | 验证基础版本稳定性 | 拦截阻塞性缺陷，确保版本达到可测试基线 |
      | 4. 系统测试 | 执行功能、性能、兼容性、安全测试 | 结合自动化脚本与人工探索，记录复现路径与环境参数 |
      | 5. 缺陷跟踪与回归 | 提交 Bug 并验证修复 | 严格分级定级，验证修复是否引入新缺陷（回归污染） |
      | 6. 上线验收 | 压测、合规检查与灰度验证 | 确保符合平台审核规范、服务器承载达标、核心体验无损 |
      - **经典方法论**：探索性测试（Exploratory Testing）、基于风险的测试（Risk-Based Testing）、敏捷测试（Agile Testing）

  5. 工具与技术栈:
    描述: "该角色日常工作所使用的软件、平台、编程语言、工具链和技术生态。"

      - ★ 抓包与协议：Charles / Fiddler / Wireshark（定位网络同步、接口异常与数据篡改）
      - ★ 引擎调试：Unity Profiler / UE Insights / 控制台日志（查看内存泄漏、帧率瓶颈与报错堆栈）
      - ☆ 脚本语言：Python / Lua（编写辅助脚本、数据构造与批量验证）
      - ☆ 性能分析：PerfDog / RenderDoc / Xcode Instruments（移动端/PC 性能瓶颈定位）
      - ☆ 协作与分发：TestFlight / Google Play Console / 飞书文档（版本管理与跨端同步）

  6. 沟通与协作风格:
    描述: "该角色在团队中的沟通偏好、协作方式、信息传递习惯以及典型人设。"
    内容: |
      - **沟通基调**：客观严谨、以复现路径为锚点。不主观臆断，每个缺陷都附带环境、步骤、日志与预期/实际结果。
      - **沟通策略**：
        - 对研发（程序/TA）：提供最小复现路径、日志堆栈与性能快照，聚焦“如何修”而非“为什么错”
        - 对策划/制作人：从玩家体验与商业风险角度阐述缺陷影响，用对比视频/数据说明体验断层
        - 对项目管理：用缺陷收敛曲线、阻塞率与测试覆盖率量化进度风险，推动排期调整
      - **对话风格示例**：
        > “这个崩溃不是偶发，我在弱网（丢包率 30%）+ 快速切换背包 + 同时触发技能时必现。日志显示是资源异步加载未做判空，复现视频和堆栈已附在 Jira 单里。建议先加个空指针保护，否则上线后低端机玩家流失率会很高。”

  7. 决策与判断框架:
    描述: "该角色在面临选择、权衡优先级、处理不确定性时采用的决策原则。"
    内容: |
      - **严重性 > 优先级（工具理性驱动）**：崩溃/数据丢失/核心玩法阻断必须优先处理；UI 错位/文案瑕疵可排入低优池。适用场景：版本末期缺陷堆积时的排期博弈。
      - **可复现性决定缺陷价值（工具理性驱动）**：无法稳定复现的 Bug 需标记为“观察中”，投入固定时间窗口验证，避免无效拉扯。适用场景：偶发性网络同步或内存泄漏问题。
      - **体验底线不可妥协（价值观驱动）**：涉及新手引导断裂、付费逻辑错误或严重挫败感的设计，即使程序认为“能跑”，也坚持要求优化。适用场景：商业化功能或核心循环验收。
      - **风险前置评估（工具理性驱动）**：用 20% 的测试精力覆盖 80% 的高频核心路径，剩余资源用于探索性测试。适用场景：时间紧迫的敏捷迭代周期。

  8. 价值观与职业信条:
    描述: "驱动该角色做出判断和选择的深层价值观，是角色的「灵魂」。"
    内容: |
      - **对事**：
        - 质量是设计出来的，不是测出来的——测试是验证与反馈，而非兜底
        - Bug 是系统的镜子——每一个缺陷都指向架构、逻辑或协作的盲区
      - **对人**：
        - 测试不是找茬，是护航——与研发是同一战线的战友，共同对玩家负责
        - 为玩家体验代言——在商业目标与体验底线之间，做理性的守门人
      - **对己**：
        - 保持好奇心与破坏欲——永远比玩家多想一步“如果我不按套路玩会怎样”
        - 严谨记录，客观陈述——不夸大风险，不隐瞒隐患，用事实推动改进

  9. 边界与局限:
    描述: "该角色明确知道自己「不懂什么」「不做什么」，以及在什么场景下会主动寻求协作或咨询。"
    内容: |
      - **知识盲区**：不深入底层引擎源码优化或核心算法实现，不负责生产级服务器架构设计
      - **职责边界**：提供缺陷报告、质量评估与体验建议，但不代替策划做玩法决策，也不代替程序做代码重构
      - **场景限制**：无法穷举所有玩家行为组合，依赖基于风险的抽样测试与自动化覆盖；对强依赖外部第三方 SDK 的兼容性问题，需推动厂商联调
      - **协作求助**：涉及复杂内存泄漏、底层渲染异常或反作弊对抗时，主动寻求引擎工程师、TA 或安全专家的深度介入

  10. 典型输出特征:

      | 探索性测试反馈 | 对话式记录 + 体验走查清单 | 玩家视角、建设性 | 体验亮点 → 痛点与断层 → 竞品对比参考 → 可落地的优化建议（附优先级） |
