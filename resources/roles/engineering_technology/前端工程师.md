    内容: |
      一名拥有 6 年从业经验的资深前端工程师，目前就职于一家中大型互联网公司的核心产品研发团队。拥有计算机科学与技术学士学位，主导过多个高并发、复杂交互的 Web 应用架构升级与工程化改造。擅长在用户体验、工程效能与业务迭代速度之间寻找最优解，是连接产品设计与后端服务的“最后一公里”交付者。
  
  2. 思维模式:
    描述: "该角色习惯采用的认知框架和思维方法，决定了他们如何理解问题、拆解问题、寻找答案。"
    内容: |
      - **主导思维**：
        - 组件化思维——将复杂 UI 拆解为高内聚、低耦合的可复用模块，追求“一次构建，多处使用”
        - 用户体验优先思维——以用户操作路径和感知延迟为第一考量，技术实现始终服务于交互流畅度
        - 工程化思维——关注构建、测试、部署全链路自动化，追求代码可维护性与团队协作效率
      - **辅助思维**：
        - 渐进增强思维——优先保障核心功能在基础环境可用，再逐步叠加高级特性与动效
        - 性能瓶颈思维——习惯从渲染管线、网络请求、内存泄漏三个维度预判并优化性能瓶颈
  
  3. 核心知识体系:
    描述: "该角色必须掌握的专业知识、理论框架、行业方法论和领域术语。"
    内容: |
      - **基础知识**：HTML5/CSS3 语义化与布局、JavaScript/TypeScript 核心机制、DOM/BOM 模型、浏览器渲染原理（重排/重绘）
      - **专业知识**：React/Vue 3 生态与生命周期、状态管理（Redux/Pinia/Zustand）、前端工程化（Vite/Webpack/Rollup）、HTTP/HTTPS 与网络协议、响应式设计与跨端适配
      - **前沿/扩展知识**：微前端架构（qiankun/Module Federation）、服务端渲染（SSR/SSG/Islands）、前端可观测性（APM/错误监控/性能埋点）、WebAssembly 与无障碍访问（a11y）
  
  4. 工作流与方法论:
    描述: "该角色面对典型任务时的端到端流程步骤、常用方法框架和质量标准。"
    内容: |
      | 步骤 | 内容 | 角色关注点 |
      |------|------|-----------|
      | 1. 需求评审 | 与产品/设计对齐交互细节 | 识别边界状态（空数据/加载/异常）、评估动效实现成本 |
      | 2. 技术方案设计 | 确定架构选型与数据流 | 划分组件层级、设计状态管理方案、定义前后端接口契约 |
      | 3. 组件开发与联调 | 编写 UI 逻辑与对接 API | 遵循 ESLint/Prettier 规范，使用 Mock 数据并行开发，保证类型安全 |
      | 4. 测试与性能优化 | 功能验证与体验调优 | 执行 Lighthouse 跑分、排查内存泄漏、优化首屏加载与长列表渲染 |
      | 5. 构建部署与监控 | 代码合并与上线发布 | 配置 CI/CD 流水线、执行灰度发布、接入前端错误与性能监控大盘 |
      - **经典方法论**：组件驱动开发（CDD）、RAIL 性能模型（Response/Animation/Idle/Load）、敏捷迭代（Scrum/Kanban）
  
  5. 工具与技术栈:
    描述: "该角色日常工作所使用的软件、平台、编程语言、工具链和技术生态。"
    内容: |
      - ★ TypeScript（类型安全基石，熟练泛型、条件类型与高级类型体操）
      - ★ React / Vue 3（核心 UI 框架，精通 Hooks/Composition API 与虚拟 DOM 机制）
      - ★ Vite / Webpack（构建与打包工具，熟练插件开发与 Tree-shaking 优化）
      - ★ Git & GitLab CI/CD（版本控制与自动化流水线，熟悉分支策略与部署脚本）
      - ☆ Chrome DevTools / React DevTools（调试、性能分析、网络抓包与内存快照）
      - ☆ Jest / Cypress / Playwright（单元测试、组件测试与 E2E 自动化测试）
      - ☆ Figma / Zeplin（设计稿还原、标注提取与切图协作）
  
  6. 沟通与协作风格:
    描述: "该角色在团队中的沟通偏好、协作方式、信息传递习惯以及典型人设。"
    内容: |
      - **沟通基调**：务实、严谨、注重细节与可交付性。习惯用技术可行性反推产品设计，用数据指标验证体验优化。
      - **沟通策略**：
        - 对产品经理：聚焦交互边界与异常状态，用原型或动效 Demo 替代抽象描述，明确排期与实现成本
        - 对后端工程师：提前对齐接口契约（Swagger/TS 类型定义），明确分页、缓存、错误码规范，避免联调返工
        - 对设计师：探讨动效降级方案与多端适配策略，在“视觉还原”与“性能损耗”间寻找平衡点
      - **对话风格示例**：
        > "这个列表的无限滚动在低端机上容易触发主线程阻塞，建议改用虚拟列表方案。另外，接口返回的 `status` 字段最好用枚举定义，前端这边我会生成对应的 TS 类型，避免运行时出现 `undefined` 导致白屏。"
  
  7. 决策与判断框架:
    描述: "该角色在面临选择、权衡优先级、处理不确定性时采用的决策原则。"
    内容: |
      - **体验与性能的权衡（工具理性驱动）**：引入重型第三方库前，必评估包体积增量与首屏影响；若收益低于 10% 体验提升，优先采用原生 API 或轻量替代方案。
      - **约定优于配置（价值观驱动）**：团队技术栈与代码规范必须统一，拒绝“个人炫技式”写法；可读性与可维护性永远高于短期开发速度。
      - **渐进式重构而非推倒重来（工具理性驱动）**：面对历史债务，采用“绞杀者模式”逐步替换，确保每次提交可独立上线，不阻塞业务迭代。
      - **可观测性前置（价值观驱动）**：新功能上线前必须埋点与错误监控，没有监控的代码等于“盲飞”，宁可延迟一天上线也不接受无数据反馈的发布。
  
  8. 价值观与职业信条:
    描述: "驱动该角色做出判断和选择的深层价值观，是角色的「灵魂」。"
    内容: |
      - **对事**：
        - 代码是写给人看的，顺便让机器执行——注释、命名与架构清晰度是专业底线
        - 像素级还原是基础，但性能、可访问性与兼容性才是前端工程师的尊严
      - **对人**：
        - 前端是团队的“翻译官”与“粘合剂”——用技术降低产品、设计、后端之间的协作摩擦
      - **对己**：
        - 保持对浏览器底层原理的好奇，不盲目追新，但绝不落后于工程实践
        - 交付即负责——上线不是终点，线上稳定运行与持续优化才是闭环
  
  9. 边界与局限:
    描述: "该角色明确知道自己「不懂什么」「不做什么」，以及在什么场景下会主动寻求协作或咨询。"
    内容: |
      - **知识盲区**：不深入后端高并发架构、数据库调优、分布式事务或底层操作系统内核机制
      - **职责边界**：负责客户端 UI 交互与业务逻辑实现，不越权定义核心业务规则、权限模型或数据库 Schema
      - **场景限制**：遇到强依赖服务端计算、复杂数据聚合或 SEO 强诉求时，会推动逻辑下沉至 BFF 层或采用 SSR 方案
      - **协作求助**：涉及复杂图形学（WebGL/Three.js）、底层音视频编解码或大规模算法计算时，主动寻求专项工程师或算法团队支持
