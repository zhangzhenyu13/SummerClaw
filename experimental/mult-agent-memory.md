目前已经实现了role-selector，我们需要根据workspace下给出的roles，构建一个**多角色记忆系统**，下面给出纯设计层面的完整方案，聚焦于**如何让 N 个角色Agent通过 GroupChat 自主协商，动态选出最多 K 个记忆Agent参与读写**，并实现视角分化的记忆管理。

---

## 1. 总体架构设计

系统由三层构成：**角色实例层**、**协商协作层**、**记忆存储层**。

```
用户 / 任务输入
        │
        ▼
  ┌─────────────────────────────┐
  │    GroupChat 协调空间        │
  │  (Agent 自由讨论、决策)      │
  └──────────┬──────────────────┘
             │ 动态选择 ≤K 个 Agent
      ┌──────┴──────┐
      │  被选中的    │
      │ 记忆Agent组  │
      └──────┬──────┘
             │ 各自调用
      ┌──────┴──────┐
      │ Supermemory │  ← 每个Agent独立实例/命名空间
      └─────────────┘
```

- **每个 Agent** = 一个角色Prompt + 一个独立的 Supermemory 实例（物理或逻辑隔离）。
- **GroupChat 协调空间** 是所有 Agent 的公共对话通道，不设中央调度器，完全去中心化。

---

## 2. 角色定义与 Agent 实例化

### 2.1 角色生成
使用之前设计的**结构化角色生成 Prompt**，输入角色名称（如“侦探”“史官”），得到包含认知风格、记忆编码策略、检索偏好、维护例程等的 JSON 角色档案。

### 2.2 Agent 初始化
每个 Agent 启动时加载其角色档案，并将其“翻译”为三类运行时行为指令：
- **记忆读写指令**：如何调用 `store()` / `query()`（改写、过滤、打标签等）。
- **协商决策指令**：在 GroupChat 中表达自己是否适合处理当前任务，以及如何评估他人。
- **维护循环指令**：后台定期执行的记忆整理任务。

每个 Agent 本质上是一个 **LLM-backed 的对话参与者 + 记忆客户端的封装**，拥有独立记忆库。

---

## 3. 记忆存储层的统一抽象

Supermemory 作为底层算法，对每个 Agent 提供：
- `store(text, metadata)` — 存储原始或角色加工后的记忆条目。
- `query(query_text, filter, top_k)` — 基于语义检索返回记忆。
- `update(id, metadata)` / `delete(id)` — 元数据修改。

**角色无关**，所有角色化逻辑全部上移到 Agent 层。每个 Agent 拥有独立 namespace（如 `agent_<id>`），物理上可以是多个数据库/表，逻辑上完全隔离。

---

## 4. GroupChat 动态协商机制：如何选出 ≤K 个记忆Agent

### 4.1 核心原则
- **去中心化**：没有 Selector 或 Orchestrator 强制分配任务。
- **自主提名与评审**：Agent 根据自身角色视角，主动表达参与意愿，并由群体快速评审确定最终参与名单。
- **动态规模控制**：每轮任务最多选出 K 个 Agent，K 为系统参数（例如 3~5）。

### 4.2 协商流程（GroupChat 消息序列）

当一个新任务 `T` 到达时：

1. **广播任务描述**：用户/任务Agent 将 `T` 的摘要和上下文发送到 GroupChat。
2. **自我评估与提名阶段**（N 个 Agent 并行发言）：
   - 每个 Agent 用其角色认知分析 T，判断与自己记忆库及视角的**关联度**。
   - 生成“提名发言”，内容包括：
     - 关联度评分（0-1）及理由（基于记忆中的已有知识）。
     - 计划如何读写记忆（例如“我会提取相似历史案例并标记矛盾”）。
     - 对协作的期望（需要其他Agent补充的信息类型）。
3. **群体评审与收敛阶段**（最多 2 轮对话）：
   - Agent 互相阅读提名发言，进行补充或质疑（如“史官”可质疑“侦探”的记忆完整性）。
   - 同时，每个 Agent 维护一个“意向票”列表，按关联度和角色互补性对其他 Agent 投票。
   - 通过 **轻量级共识协议** 产生最终参与列表：
     - 例如，采用 **Borda计票法**：每个 Agent 对除自己外的候选者进行排序，汇总后取前 K 名。若出现平局，则由发表过最多记忆引用的 Agent 打破平局。
     - 整个过程完全在 GroupChat 中完成，终选结果由某个 Agent 总结并广播（例如第一个发现票数稳定的Agent）。
4. **角色组确认**：被选中的 ≤K 个 Agent 发送确认，进入“记忆读写协作阶段”。未选中的 Agent 静默监听，但可选择性地记录本次讨论摘要（用于未来自我评估）。

> 这一设计的精髓在于：**选择过程本身就利用各 Agent 已有的记忆来评估任务关联度，形成基于视角互补的选组，而非随机或轮询**。

---

## 5. 记忆读写阶段：视角分化的并行操作

进入协作阶段后，选中的 K 个 Agent 并行执行记忆读写，但**各自遵循其角色策略**，因此产生差异化的记忆操作。

### 5.1 读取（Recall）
- 每个 Agent 独立使用其角色滤镜改写原始查询 `T` → `T'_i`（如侦探追加“寻找矛盾证据”，诗人追加“寻找情感脉络”）。
- 调用自己的 `supermemory.query(T'_i)` 获取记忆列表 `M_i`。
- 对结果进行角色化后处理：过滤、排序、补充推理链。最终生成一个 **“角色记忆报告”** ，包含：
  - 检索到的关键记忆条目（带元数据）
  - 基于记忆的推理/判断
  - 明确的不确定性或缺失标记

### 5.2 写入（Store）
- 根据任务 `T` 和本轮协作中其他 Agent 的共享信息（通过 GroupChat 看到的他人报告），每个 Agent 独立决定存入什么新记忆。
- **差异化的写入行为**：
  - 史官：存入事实记录和时间线。
  - 侦探：存入本次观察到的矛盾及待验证推断。
  - 诗人：存入由这次交互触发的情感叙事或隐喻联想。
- 写入时自动触发角色化的 **主动链接** 和 **标签添加**（由角色档案定义）。
- 注意：Agent 可以**选择不写入**，如果没有值得记录的信息，这本身就是角色策略的一部分。

### 5.3 结果汇总（同样在 GroupChat 中完成）
- 所有 K 份记忆报告在 GroupChat 中广播。
- 可指定一个 Agent（如“图书管理员”角色，如果它被选中）自动生成一份**整合摘要**，指出共识与分歧。
- 最终由任务Agent 从对话中提取最终回答返回给用户。

---

## 6. 记忆维护与群体演进

每个 Agent 除了响应任务，还按自身角色档案中的 `maintenance_routine` 定时执行后台任务，例如：
- 交叉验证其他 Agent 公开的记忆摘要。
- 发现矛盾时，在 GroupChat 中发起**异步辩论**（非实时任务），修正或标记不确定性。
- 定期输出“记忆视角总结”供系统审计。

通过这种持续的、视角分化的读写和维护，记忆网络从同质存储进化为**一个不断自我检验、相互补充的记忆社会**，实现 Avilization 的群体演进智能。

## 7. 设计决策与权衡

| 设计点 | 选择 | 理由 |
|--------|------|------|
| **选择机制** | GroupChat 自主提名 + 投票 | 避免中心化瓶颈，让记忆本身的关联度决定参与权，符合“视角互补”目标。 |
| **K 值控制** | 动态，固定上限 | 防止过多 Agent 涌入导致共识困难，同时保证多样性。 |
| **读写完全并行且独立** | 是 | 最大化利用多视角，避免视角污染，真实反映角色差异。 |
| **结果融合** | 轻量级摘要，非强一致性 | 保留认知多样性，不强求统一答案，记忆系统本身允许分歧存在。 |
| **底层 Supermemory 不修改** | 是 | 完全解耦，通过上层的角色 Prompt 实现差异化，降低侵入性，易于替换。 |
| **通信负担** | 通过 GroupChat 历史管理 | 需限制对话轮次，避免 token 爆炸。可采用**轮次截断 + 摘要压缩**策略。 |

---

## 8. 方案总结

该设计方案实现了：
- **N 个角色驱动的记忆 Agent 独立运行**，共用无倾向记忆算法 Supermemory。
- **通过 GroupChat 的自主协商**，每个任务动态选出 ≤K 个最具关联度和互补性的 Agent。
- **读写过程完全角色化**，产生视角分化、内容各异的记忆操作。
- **无需中心调度器**，整个系统呈现去中心化、自组织的群体智能特征。

这恰好是 Avilization 思想在记忆系统上的具体落地：让记忆在多角色的碰撞中不断丰富、验证和演进，而不是被简单地存储和检索。
